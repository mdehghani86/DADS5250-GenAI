Think about any startup. One person does everything -- sales, engineering, marketing, support. It does not scale. The moment you hire specialists and give each one a clear role, everything changes. Multi-agent systems work the same way.

In Module 6, you built a single agent. It could use tools, reason about problems, take actions. Impressive. But a single agent has limits. Ask it to research a topic, write a report, fact-check the report, and format it for publication -- and it starts losing context. It forgets earlier steps. It gets confused about what role it is playing.

The solution is not a smarter agent. It is a team of agents. One researcher. One writer. One editor. Each with a clear role, clear tools, and a clear goal. They hand work to each other like colleagues in a company.

This is what CrewAI does. It lets you define agents with roles, give them tasks, and orchestrate how they work together. You become the manager. The agents become your team.

By the end of this module, you will have a working multi-agent crew that can research, write, and review -- all without your intervention.

CrewAI is a Python framework for building multi-agent systems. Think of it as an org chart for AI.

Three core concepts. First, Agents. Each agent has a role, a goal, and a backstory. The role tells the agent what it is -- a researcher, a writer, a reviewer. The goal tells it what success looks like. The backstory gives it context and personality.

Second, Tasks. Each task is a specific piece of work with a description, an expected output, and an assigned agent. Tasks can depend on each other -- the writer cannot start until the researcher finishes.

Third, Crews. A crew is the team. You assign agents and tasks to a crew, choose a process type -- sequential or hierarchical -- and kick it off. Sequential means tasks run one after another. Hierarchical means a manager agent delegates work.

The code is surprisingly simple. You define your agents in a few lines. You define your tasks. You create a crew, call kickoff, and the agents start working.

Delegation is where it gets interesting. Agents can ask other agents for help. The researcher might ask the writer to clarify a question. The manager might reassign a task if an agent is stuck.

Now that you understand the basics, let us talk about how agents actually communicate.

In a sequential crew, the output of one task becomes the input to the next. The researcher produces findings. Those findings automatically flow into the writer's task as context. The writer produces a draft. That draft flows into the reviewer.

Task dependencies let you control this flow. You can say: this task depends on that task. CrewAI handles the rest -- making sure tasks run in the right order and passing results along.

Memory is another powerful feature. Agents can remember what happened in previous tasks. Short-term memory within a run, and long-term memory across runs. This means your crew gets better over time.

For complex workflows, the hierarchical process is powerful. You designate one agent as the manager. The manager reads all the tasks, decides which agent should handle each one, reviews the output, and can send work back for revision.

This is where multi-agent systems start to feel like real organizations.

Here is what you accomplished this week. You went from building single agents to building teams of agents. You defined roles, assigned tasks, and orchestrated workflows.

Think about what that means. You are no longer writing code that does one thing. You are designing systems where multiple AI agents collaborate to solve problems that are too complex for any single agent.

The researcher finds information. The writer turns it into content. The reviewer catches errors. Each agent is focused. Each agent is good at one thing. Together, they produce work that no single agent could match.

Why Multi-Agent?

Single agents are powerful. But they hit a ceiling. Multi-agent systems break through it.

The Limits of a Single Agent

A single agent has one context window, one set of instructions, and one persona. Ask it to be a researcher AND a writer AND an editor, and it gets confused. It loses track of which role it is playing. The context window fills up. Quality degrades.

This is not a model problem -- it is an architecture problem. The solution is specialization.

The Multi-Agent Advantage

Specialization

Each agent focuses on one job. A researcher researches. A writer writes. Quality goes up because each agent has a narrow, clear role.

Scalability

Need a new capability? Add an agent. No need to retrain or redesign the whole system. Just plug in a new specialist.

Reliability

One agent can check another's work. A reviewer catches errors the writer missed. Built-in quality control.

Real-World Examples

  • Content pipeline: Researcher gathers data, Writer drafts article, Editor polishes, Publisher formats for distribution
  • Code review: Developer writes code, Tester generates tests, Reviewer checks quality, Documenter writes docs
  • Customer support: Classifier routes tickets, Specialist handles domain questions, Escalator flags complex issues to humans

Agent Roles & Workflows

A visual guide to the most common multi-agent patterns.

Pattern 1: Sequential Pipeline

Researcher
→
Writer
→
Reviewer

Each agent completes its task before the next starts. Output flows downstream. Best for linear workflows where each step depends on the previous.

Pattern 2: Hierarchical (Manager)

Manager
↓
Researcher
↓
Writer
↓
Reviewer

A manager agent delegates tasks, reviews outputs, and can reassign work. Best for complex projects where task order is dynamic.

Pattern 3: Peer Review

Agent A
↔
Agent B

Agents review each other's work and iterate. Best for quality-critical outputs where multiple perspectives improve the result.
