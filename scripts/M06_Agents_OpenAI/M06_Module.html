I play chess occasionally. When you are new to the game, you move one piece and then you wait. You react to whatever your opponent does.

But experienced players do not think one move ahead. They think five or six moves ahead. They plan contingencies. They adapt when something unexpected happens.

That is the difference between a chain and an agent.

A chain follows a fixed sequence -- step one, step two, step three, every time. An agent receives a goal, reasons about what needs to happen, selects the right tool, observes the result, and decides what to do next. The process is dynamic. Each action informs the next one.

After this module, you will build AI systems that think like experienced chess players -- not beginners.

Let us define what an agent actually is.

An agent is an AI system that reasons, plans, and takes actions autonomously. It does not just generate text -- it decides what to do, does it, observes the result, and decides what to do next.

The core pattern is called ReAct -- Reason, Act, Observe, Repeat.

Here is how it works. The agent receives a question or task. It reasons about what it knows and what it needs. It selects a tool -- maybe a calculator, maybe a search engine, maybe a database query. It calls that tool. It reads the result. Then it asks itself: am I done? If not, it reasons again and picks the next action.

Tools are functions the agent can call. Think of them as capabilities -- search, calculate, look up data, read a file. The agent sees a list of available tools with descriptions, and it decides which one is appropriate for the current step.

The agent does not use all tools every time. It reads the question, reasons about which tool is appropriate, calls it, and uses the result to decide what to do next.

Here is the key distinction. A chain is a script. An agent is a decision-maker. A chain does the same thing every time. An agent adapts to the situation.

Let us talk about tools -- the capabilities that make agents powerful.

A tool is simply a function that the agent can call. A calculator. A web search. A database query. A file reader. Anything you can write as a Python function, you can give to an agent as a tool.

The agent sees a list of available tools -- each with a name, a description, and the parameters it accepts. When the agent reasons about a problem, it decides WHICH tool to use and with WHAT arguments.

You already learned function calling in Module 3. That was one call -- you send a message, the model says "call this function with these arguments," and you execute it once.

Agents use function calling in a LOOP. They call a tool, read the result, reason about it, call another tool, read that result, and continue until the task is complete. That loop is what makes agents fundamentally different from simple function calling.

The power is in the combination. An agent with five well-designed tools can handle hundreds of different tasks -- because it decides at runtime which tools to use and in what order.

Now let us talk about LangGraph -- a framework for building agent workflows as graphs.

LangGraph is built on top of LangChain, but it takes a fundamentally different approach. Instead of chains -- which are linear -- LangGraph lets you build workflows as directed graphs.

Three concepts. Nodes are steps -- an LLM call, a tool execution, a decision function. Edges are transitions -- what happens after each step. State is shared memory that flows through the entire graph.

Think of it as a flowchart that the AI navigates at runtime. You define the possible paths. The agent decides which path to take based on the current state.

Here is a simple example. Imagine a graph with three nodes: classify the user's input, route it to the right handler, and generate a response. The classify node reads the input and adds a category to the state. The routing edge checks that category and sends the state to one of several response nodes. Each node reads from the state and writes back to it.

This is the mental model. Nodes process state. Edges move state between nodes. The graph as a whole represents every possible path your agent can take.

Let us walk through how you actually build a LangGraph agent.

Step one: define the state. This is a TypedDict or dataclass that describes what information flows through your graph. Messages, tool results, intermediate decisions -- everything lives in the state.

Step two: create nodes. Each node is a Python function that takes the current state, does something with it, and returns an updated state. One node might call the LLM. Another might execute a tool. Another might make a routing decision.

Step three: add edges. Normal edges connect one node to the next unconditionally. Conditional edges check the state and route to different nodes based on what they find. This is where the decision-making happens.

Step four: compile and run. You call compile on the graph to get a runnable object. Then you invoke it with an initial state, and LangGraph handles the rest -- moving through nodes, following edges, updating state at each step.

LangGraph gives you fine-grained control over agent behavior. Every decision point is explicit in the graph. You can see exactly where branching happens, what conditions trigger different paths, and how the state evolves. That visibility is what makes LangGraph powerful for production agent systems.

When to Use Which

Use a chain when you have a predictable workflow with known steps -- summarize a document, translate text, extract data. The path is always the same.

Use an agent when the task is open-ended -- research a topic, debug a problem, answer a question that might require multiple tools. The agent figures out the path at runtime.

The ReAct Pattern

The reasoning loop that powers every modern AI agent.

The ReAct Loop
Reason
"What do I know? What do I need?"
Act
"Call the appropriate tool"
Observe
"Read the result"
Repeat until task is complete

Concrete Example

User asks: "What is the weather in Boston and should I bring an umbrella?"

REASON

"I need current weather data for Boston. I should call the weather API."

ACT

Calls get_weather("Boston, MA")

OBSERVE

Result: "Partly cloudy, 62F, 40% chance of rain in afternoon"

REASON

"40% chance of rain -- I should recommend bringing an umbrella just in case."

RESPOND

"It is 62F and partly cloudy in Boston with a 40% chance of afternoon rain. I would recommend bringing an umbrella."

The 2026 Agent Landscape

How agent building evolved from framework complexity to batteries-included SDKs -- and why LangGraph still matters.

The Evolution

2024-2025 -- THE FRAMEWORK ERA
Pick framework, define tools, write glue code
LangChain agents, AutoGen, custom ReAct loops. Lots of boilerplate. Every team built their own patterns.
2026 -- THE SDK ERA
Batteries-included SDKs from model providers
Claude Agent SDK, OpenAI Agents SDK, Google ADK. Built-in tool use, memory, and orchestration. Much less code.

Why LangGraph Still Matters

🛠

Complex Orchestration

Multi-step workflows with branching logic

🔌

Custom Logic

Fine-grained control over every decision point

🤖

Multi-Model

Mix different LLMs in one workflow

When to Use What

A decision guide for choosing the right approach for your AI system.

Scenario Best Tool
Fixed, predictable workflow Chain (LangChain)
Dynamic, tool-using AI Agent (LangGraph)
Quick prototype OpenAI Agents SDK
Complex multi-agent CrewAI (M08)
Production with built-in tools Claude Agent SDK (M12)

The Key Insight

There is no single best framework. The right choice depends on your use case. Chains are perfect for predictable workflows. Agents shine when the task requires reasoning. SDKs are fastest for prototyping. Multi-agent systems handle complex orchestration.

This course teaches you all of them -- so you can pick the right tool for each situation.
