Video 2: "What Is an Agent?"

Voiceover + animation -- 7-8 minutes -- 3 segments
SEG A Defining an Agent
Voiceover + Animation ~3 min

An agent is an AI system that reasons, plans, and takes actions autonomously. Let me break that down.

A chain runs the same steps every time. An agent decides what to do at each step. It looks at the current situation, reasons about what information it needs, picks a tool to get that information, observes the result, and then decides what to do next. It keeps going until the task is done.

The key distinction: a chain is a script. An agent is a decision-maker.

Animation
Comparison Animation Chain: linear arrows, A to B to C, same every time. Agent: branching flowchart, decisions at each node, different paths light up on different runs. ~12 sec
SEG B The ReAct Loop
Voiceover + Animation ~3 min

Every modern agent runs on the same loop. It is called ReAct -- Reason, Act, Observe.

First, the agent reasons. "What do I know? What do I need?" Then it acts -- it calls a tool. A search, a calculator, a database query. Then it observes the result. "What did I learn?" And then it reasons again. "Is the task done? Do I need another step?" The loop continues until the agent decides it has enough information to respond.

Let me give you a concrete example. A user asks: "What is the weather in Boston, and should I bring an umbrella?" The agent reasons: I need weather data. It acts: calls the weather API. It observes: 62 degrees, 40 percent chance of rain. It reasons again: 40 percent rain means bring an umbrella. Done.

That is three lines of reasoning, one tool call, and a complete answer. A chain could not do this -- it would need the weather tool hardcoded into a fixed pipeline. The agent decided to use it.

Animation -- ReAct Loop
Circular Animation Brain icon (Reason) -- hand icon (Act) -- eye icon (Observe) -- loops back. Each cycle adds information. The Boston weather example plays out concretely in the loop: Reason step shows thinking text, Act step shows API call, Observe step shows result. ~20 sec
MEDIA TEAM: Use blue for Reason, green for Act, amber for Observe. The loop should feel like a thinking process. Show the concrete example playing out step by step inside the loop, not just abstract icons.
SEG C When to Use Agents
Voiceover + Animation ~2 min

So when do you use an agent versus a chain?

Use a chain when you have a predictable workflow with known steps. Summarize a document. Translate text. Extract data from a form. The path is always the same, and that is fine.

Use an agent when the task is open-ended. Research a topic -- you do not know how many searches it will take. Debug a problem -- you do not know which files to check. Answer a question that might require multiple tools in an unpredictable order. The agent figures out the path at runtime.

Chains are efficient and predictable. Agents are flexible and smart. You want both in your toolkit.

Video 3: "Tools: Teaching AI to Use Functions"

Voiceover + animation -- 5-6 minutes -- 2 segments
SEG A What Is a Tool
Voiceover + Animation ~3 min

A tool is a function the agent can call. That is it. A Python function with a name, a description, and parameters.

The agent sees a list of available tools with their descriptions. When it decides it needs information, it picks a tool, fills in the arguments, and calls it. The result comes back, and the agent uses it to continue reasoning.

You already learned function calling in Module 3. One call, one response. Agents use function calling in a loop -- chaining multiple tool calls until the task is complete. That is the difference. Same mechanism, repeated intelligently.

Video 4: "Introduction to LangGraph"

Voiceover + animation -- 6-7 minutes -- 3 segments
SEG A Why LangGraph
Voiceover + Animation ~2 min

You understand agents. You understand tools. Now you need a framework to build them properly. That framework is LangGraph.

LangGraph models your agent workflow as a graph. Nodes are steps -- LLM calls, tool calls, decisions. Edges are transitions -- what happens next. And state is shared memory that flows through the entire graph.

Think of it as a flowchart that the AI navigates at runtime. You design the flowchart. The AI decides which path to take.

Animation -- LangGraph
Interactive Flowchart Flowchart builds itself: nodes appear one by one (boxes), edges connect them (arrows). Data flows through, nodes light up as activated. Conditional edges show branching. State updates visible at each node as a small data panel. ~15 sec
MEDIA TEAM: This is the key visual for LangGraph. Nodes should be clean boxes with labels. Conditional edges should show a diamond decision point. Use the blue accent color for active nodes, gray for inactive. Data flow should be animated as a glowing line traveling along edges.
SEG B Nodes, Edges, State
Voiceover + Animation ~2 min

Let me define the three building blocks.

Nodes are functions. Each node takes the current state, does something -- calls the LLM, runs a tool, makes a decision -- and returns updated state. That is it. A function in, updated state out.

Edges connect nodes. A normal edge says "after this node, always go to that node." A conditional edge says "after this node, look at the state and decide where to go." This is where the intelligence lives -- the graph makes different decisions based on the data.

State is a dictionary that every node can read and write. It is how nodes communicate. The LLM node writes its response to state. The tool node reads the response and writes its result. Everything flows through state.

Animation Spec
Three-Part Animation Part 1: Node as a box, function goes in, state comes out. Part 2: Normal edge (straight arrow) vs conditional edge (diamond + branching arrows). Part 3: State as a flowing dictionary panel moving along edges between nodes.
SEG C LangGraph in Context
Voiceover + Animation ~2 min

Why LangGraph when model providers now ship their own agent SDKs?

In 2024, you had to pick a framework -- LangChain agents, AutoGen, custom ReAct loops. Lots of boilerplate. In 2026, Claude Agent SDK, OpenAI Agents SDK, and Google ADK handle most simple agent tasks out of the box.

But LangGraph still matters for complex orchestration. When you need multi-step workflows with branching logic. When you need fine-grained control over every decision point. When you want to mix different LLMs in one workflow. LangGraph gives you that control. We will cover the newer SDKs in Module 12.

Video 5: "Building with LangGraph"

Voiceover + code display -- 5-6 minutes -- 2 segments
SEG A The 5-Step Pattern
Voiceover + Code ~3 min

Building a LangGraph agent follows five steps. Every time.

Step one: define the state. What information flows through the graph? Messages, tool results, decisions -- all stored in a typed dictionary.

Step two: create nodes. Each node is a Python function that takes state and returns updated state.

Step three: add edges. Connect your nodes. "After classify, go to respond."

Step four: add conditional edges. "After the LLM responds, check: did it want to call a tool? If yes, go to the tool node. If no, go to the end." This is the decision-making.

Step five: compile and run. Call graph.compile() to get a runnable object. Invoke it with your initial state.

Animation
Step-by-Step Build Five numbered steps building a graph visually: (1) state schema appears, (2) node boxes appear, (3) arrows connect them, (4) a diamond decision point adds branching, (5) a "compile" button turns everything into a runnable agent. ~15 sec
SEG B The Code
Voiceover + Code ~2 min

Here is a minimal LangGraph agent.

from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated

class State(TypedDict):
    messages: Annotated[list, "append"]

graph = StateGraph(State)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tools)

graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_use_tool,
    {"yes": "tools", "no": END})
graph.add_edge("tools", "agent")

app = graph.compile()

The conditional edge is the heart of it. After the agent responds, we check: does it want a tool? If yes, route to tools. If no, finish. The tools node runs the function and routes back to the agent. That loop is the ReAct pattern -- expressed as a graph.
