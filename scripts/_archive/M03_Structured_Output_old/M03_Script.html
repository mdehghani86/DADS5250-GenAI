Video 2: "JSON Mode & Function Calling"

Voiceover + animation/code -- 10-12 minutes -- 6 segments
SEG A Why Structure Matters
Voiceover + Animation ~2 min

Up until now, every response from the API has been text. A string. A paragraph. And text is great for humans -- but terrible for code.

Code needs data. Structured data. Fields with names. Values with types. Lists with consistent formats. That is JSON -- JavaScript Object Notation. And it is the language of every API, every database, every web application on earth.

JSON mode forces the model to return valid JSON -- guaranteed parseable, guaranteed structured. No more hoping the model gives you the right format. You demand it.

SEG A2 Live Comparison: Text vs JSON
Voiceover + Code ~3 min

Let me show you the difference. Same prompt, two modes.

WITHOUT JSON mode. Prompt: "List 3 programming languages with their primary use case."

Response: "Sure! Here are three popular programming languages: 1. Python -- Python is widely used for data science, machine learning, and web development. It is known for its readability..." And it continues for 200 words.

Try parsing that with code. You would need regex, string splitting, error handling -- and it would break the moment the model changes its formatting.

WITH JSON mode. Same prompt, plus response_format equals json_object.

Response: A clean JSON object. Languages as an array. Each with a name and use_case field. Three items. Done.

json.loads() and you have a Python dictionary. That is the difference between a demo and a product.

SEG B Pydantic Validation
Voiceover + Code ~2 min

JSON mode guarantees valid JSON. But it does not guarantee the right structure. What if the model uses "total_amount" instead of "total"? What if it returns a string where you need a number?

Pydantic solves this. You define a Python class that describes exactly what fields you expect, what types they should be, and which are required. Then you pass the model's JSON through it. If everything matches, you get a clean validated object. If anything is wrong, you get a clear error.

SEG B2 Pydantic Catching Bugs
Voiceover + Code ~2 min

Here is why Pydantic matters. The model returned this JSON for an invoice:

invoice_num instead of invoice_number. Total as a string with a dollar sign, not a float. Items as a comma-separated string, not a list.

Looks fine to a human. But three problems that would break your pipeline.

Without Pydantic, this goes into your database and something breaks downstream -- maybe tomorrow, maybe next week. A silent bug.

With Pydantic, you define the schema. Run the JSON through it. Pydantic catches all three errors immediately. Field invoice_number missing. Value is not a valid float. Value is not a valid list.

Three bugs caught before they touched your database. That is Pydantic.

SEG C Function Calling Concept
Voiceover + Diagram ~2 min

Function calling is one of the most important concepts in this course -- because it is the bridge between a chatbot and an agent.

A chatbot generates text. An agent takes actions. Function calling is what makes the difference.

You define tools -- functions with names, descriptions, and parameter schemas. The model reads the user's request and decides: which tool should I use? What arguments should I pass?

The model returns a structured JSON object. Your code executes it and feeds the result back. This loop -- reason, call, observe, repeat -- is the foundation of everything we build from Module 6 onward.

SEG C2 Function Calling Live Example
Voiceover + Code ~3 min

Let me walk through a complete example.

Step 1 -- Define the function schema. Name: get_weather. Description: Get current weather for a city. Parameters: city as a string, required. Unit as a string, either celsius or fahrenheit.

Step 2 -- Send it with the user's message. User says: "What is the weather like in Boston right now?"

Step 3 -- The model responds -- not with text, but with a function call. Name: get_weather. Arguments: city equals Boston, unit equals fahrenheit.

The model decided: this question needs the weather function. The city is Boston. The user probably wants Fahrenheit.

Step 4 -- Your code executes the function, calls a real weather API, gets the actual data.

Step 5 -- You send the result back. The model generates a natural language response: "It is currently 62 degrees Fahrenheit in Boston with partly cloudy skies."

The model did not know the weather. It knew it needed a tool. It chose the right tool and the right arguments. That is function calling. And in Module 6, we will build full agents that chain multiple function calls together.
