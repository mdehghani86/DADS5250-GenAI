Video 2: "What Are LLMs?"

Voiceover + animation -- 6-8 minutes -- 3 segments
SEG A What Is a Large Language Model
Voiceover + Animation ~3 min

LLM stands for Large Language Model. Here is what that means.

Take a significant portion of human knowledge -- books, code, research, conversations. Compress it into a mathematical model. The model does not store the text word-for-word. It learns patterns.

When you give it a prompt, it uses those patterns to generate something new. Every response is original. Every piece of code is composed, not retrieved. That is generative AI -- and LLMs are the engine behind it.

SEG B Platform vs API
Voiceover + Animation ~2 min

You can access these models two ways.

The first is through a platform -- ChatGPT, Gemini, Claude. You open a browser, type a message, get a response. Think of it as walking into a restaurant and ordering at the counter. It works. It is convenient.

The second way is through an API. Your Python code sends a request directly to the model. Think of this as building the delivery app -- you handle thousands of orders, route them, process them, all automatically. Same kitchen, same food, completely different power.

Storyboard -- Platform vs API

SEG C How LLMs Generate Text
Voiceover + Animation ~2 min

Here is something worth understanding. LLMs do not think. They predict.

When you send a prompt, the model looks at all the words so far and calculates -- what is the most likely next word? It picks one, adds it, and repeats. Word by word. Token by token.

This is why temperature matters -- at zero, it always picks the most likely word. At higher values, it takes risks. And this is why LLMs sometimes hallucinate -- they generate text that sounds right but is factually wrong.

The model does not know what is true. It knows what is likely.

Video 3: "What Is an API?"

Voiceover + code display -- 5-6 minutes -- 2 segments
SEG A API from Zero
Voiceover + Animation ~3 min

API stands for Application Programming Interface. Let me make that simple.

Think of a vending machine. You put something in -- a coin and a button press. Something happens inside that you cannot see. And something comes out -- your drink.

You do not need to know how the machine works inside. You just need to know two things: what to put in, and what comes out.

An LLM API works the exact same way. You put in a prompt -- your question. Something happens on the model provider's servers -- the model thinks. And out comes a response -- the answer.

And here is the important part: every major LLM has an API. OpenAI has one. Google has one for Gemini. Anthropic has one for Claude. Meta's Llama, Mistral, xAI's Grok -- all of them. Different companies, different models, but the same idea. Once you know how to use one API, you can use them all. The pattern is the same: send a prompt, get a response.

SEG B The Code
Voiceover + Code ~2 min

Here is what it looks like in code.

from openai import OpenAI
client = OpenAI()

response = client.chat.completions.create(
    model='gpt-4.1-mini',
    messages=[
        {'role': 'system', 'content': 'You are a helpful assistant.'},
        {'role': 'user', 'content': 'What is an API?'}
    ]
)

print(response.choices[0].message.content)

Seven lines. That is all it takes. Seven lines and you have programmatic access to one of the most powerful AI models on earth.

Let me show you what happens when you run it. You hit enter. About one second later, the output appears in your terminal:

>>> Output:
An API, or Application Programming Interface, is a set of rules
and protocols that allows different software applications to
communicate with each other. It defines the methods and data
formats that applications can use to request and exchange
information...

That is it. A real response, from a real model, running on real servers -- triggered by your seven lines of code.

Now imagine running those seven lines ten thousand times in a loop. Or connecting them to a database. Or triggering them from a web application. Or swapping the model name from gpt-4.1-mini to gemini-2.5-flash or claude-sonnet -- same pattern, different provider. That is the power of an API.

Storyboard -- Three Dials
MEDIA TEAM: Three tiles fade in together as Prof says "temperature, tokens, and cost". Hold ~3 seconds. Then the next segment dives into each one individually.
SEG B Temperature
Voiceover + Animation ~60 sec

First: temperature. Temperature controls randomness.

At zero, the model gives you the same answer every time -- the most predictable, most likely response. Ask it the same question twice, you get the same reply twice. This is what you want for tasks like data extraction, classification, or anything where you need repeatable output.

At one, the model takes more risks. Some creativity, some variation. This is the sweet spot for most conversational tasks.

At two, the model gets very creative -- sometimes too creative. Useful for brainstorming, poetry, or ideation. Less useful when you need accuracy.

Rule of thumb: if you want reliability, go lower. If you want variety, go higher.

Storyboard -- Temperature Dial
MEDIA TEAM: Slider slides from 0 → 1 → 2 synced to narration. As the slider lands on each value, the corresponding sample-output card below lights up. Show the actual contrast: temp=0 gives the same answer 3 times, temp=2 gives 3 wildly different answers.
SEG C Tokens
Voiceover + Animation ~60 sec

Second: tokens. Tokens are how LLMs measure text.

A token is roughly four characters, or about three-quarters of a word. The word "building" might be one token, or it might be split into "build" and "ing" -- depending on the tokenizer. Short common words are usually one token. Long or rare words often split into two or three.

Here is why this matters. When you send a prompt, you pay for input tokens. When the model responds, you pay for output tokens. Both counts show up on your bill. The model also has a maximum context window -- the total number of tokens it can handle in one call. Go over, and the request fails.

So tokens are two things at once: the unit of measurement, and the unit of billing.

Storyboard -- Token Chips
MEDIA TEAM: "Building with AI." types into view first. After a beat, it splits into colored token chips one at a time with a soft click sound. Counter ticks 1...2...3...4...5. Then the "Tokens are both..." dual panel fades in showing measurement vs billing.
SEG D Cost
Voiceover + Animation ~45 sec

Third: cost. Here is the good news. This technology is remarkably cheap.

GPT-4.1 Mini, the model you will use for most of this course, costs about fifteen cents per million input tokens. A million tokens is roughly seven hundred and fifty thousand words -- more than ten full-length novels. For fifteen cents. Output tokens are a bit more expensive, but the total cost for most labs in this course will land around a few cents per run.

That changes how you think about building. You do not have to be stingy. You can run a prompt a thousand times while testing. You can let your script iterate. Cost is rarely the limiting factor anymore -- time and design are.

You will experiment with all three of these -- temperature, tokens, and cost -- in Lab 2.
