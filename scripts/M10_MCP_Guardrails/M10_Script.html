Video 1: "The Spark" -- The USB of AI

Prof on camera -- 3-4 minutes -- 3 segments
KEY MESSAGE Connection without safety is a liability, not a feature.
SEG 01 The USB Analogy
On Camera ~60 sec Medium close-up

Think about USB. Before USB, every device had its own proprietary connector. Printers, cameras, keyboards. All different cables. USB created one standard.

MCP, the Model Context Protocol, is doing the same thing for AI. One standard way for any AI model to connect to any tool, any database, any service.

Storyboard -- USB → MCP
MEDIA TEAM: Two-phase animation. Phase 1: tangled cable mess morphs into single USB cable. Phase 2: tangled API integrations morph into single MCP protocol. Same visual rhythm, different domain. Keep it clean and minimal.
SEG 02 Before MCP
On Camera ~60 sec

Before MCP, every integration was custom. Want your AI to read from a database? Write a custom tool. Want it to search the web? Write another tool. Want it to send emails? Another tool. Every integration, from scratch.

MCP changes that. It defines a standard protocol. A server exposes capabilities, a client connects to them. Any AI model can connect to any MCP server. OpenAI adopted it. Google adopted it. Anthropic created it. The entire industry is converging on this one standard.

Storyboard -- The MCP Hub
MEDIA TEAM: Clean architecture diagram. Left: AI model icons (Claude, GPT, Gemini). Center: MCP protocol hub glowing blue. Right: database, email, web, file system icons. Lines connect through the hub. Animate the discovery: tools appear and register, AI sees them light up.
SEG 03 Connection + Safety
On Camera ~40 sec

But connection without safety is a liability, not a feature. If your AI can read from a database, it can also leak data. If it can send emails, it can send harmful ones. If it can execute code, it can execute malicious code.

That is why this module covers both MCP and guardrails. Connection and safety. Power and responsibility. You need both.

Video 2: Building MCP Servers

Voiceover + code walkthrough -- 8-10 minutes -- 3 segments
SEG A MCP Architecture
Voiceover + Diagram ~3 min

An MCP server is a program that exposes tools to AI models through a standard protocol. Here is how to build one.

The architecture is simple. You have a server that registers tools. These are functions with names, descriptions, and parameter schemas. You have a client, your AI application, that discovers those tools and calls them.

Storyboard -- MCP Server / Client Architecture
SEG B Building with the MCP SDK
Voiceover + Code ~4 min

In Python, you use the MCP SDK. You create a server, decorate functions as tools, and run it. The SDK handles the protocol. The JSON-RPC messages, the capability negotiation, the error handling.

Here is what makes this powerful. Once you build an MCP server, any MCP client can use it. Claude Desktop, Cursor, your own applications. They all speak the same protocol. Build once, connect everywhere.

Storyboard -- Build an MCP Server in ~15 Lines

In the lab, you will build an MCP server that connects AI to a SQLite database. The AI will be able to query tables, run SQL, and get results. All through MCP. No custom tool code. Just the standard protocol.

Video 3: Implementing Guardrails

Voiceover + diagrams -- 6-8 minutes -- 4 segments
SEG A Three Layers of Safety
Voiceover + Animation ~2 min

Guardrails are the safety systems that keep AI applications from going off the rails. There are three layers you need.

Layer one: input validation. Before any user input reaches the LLM, you validate it. Check for prompt injection attacks, someone trying to override the system prompt. Check for prohibited content. Check for excessive length. This is your first line of defense.

Storyboard -- Three Layers of Guardrails
MEDIA TEAM: Concentric circles/shields. Outer ring = input validation (blocks bad inputs). Middle ring = prompt injection detection. Inner ring = output filtering. Core = PII redaction. Animate threats hitting each layer and being caught. Red particles blocked, green particles pass through.
SEG B Output Filtering
Voiceover + Code ~2 min

Layer two: output filtering. After the LLM generates a response, you check it. Does it contain harmful content? Does it reveal system prompts or internal data? Does it include hallucinated facts that could be dangerous? You filter before the user sees it.

Storyboard -- Output Filtering Pipeline
SEG C Behavioral Constraints
Voiceover + Diagrams ~2 min

Layer three: behavioral constraints. These are the rules your AI follows. It will not execute arbitrary code. It will not access data outside its scope. It will not make claims about topics it should not discuss. These constraints are embedded in the system prompt and enforced in code.

Storyboard -- Real-World Incidents
MEDIA TEAM: Clean card layout. Each incident: headline, what happened, what guardrail was missing. Red border for "what went wrong," green border for "how to prevent." Two cards, shown sequentially.
SEG D Frameworks
Voiceover + Code ~2 min

The frameworks make this easier. Guardrails AI, NeMo Guardrails from NVIDIA, and OpenAI's moderation API all provide pre-built safety checks. In the lab, you will implement all three layers.
