I was at a symphony concert last year. Dozens of musicians on stage, each playing a completely different instrument. Individually, each one is talented. But without the conductor standing at the front, coordinating every section, every tempo change, every entrance -- it would just be noise.

LangChain is that conductor for your AI applications.

Until now, you have been making single API calls. One prompt in, one response out. But real AI applications are rarely that simple. You might need the model to remember what the user said three messages ago. You might need it to search a database before responding. You might need it to call a tool, format the output, and pass it to a second model.

LangChain gives you pre-built components for every part of an AI pipeline -- and a standard way to connect them together.

LangChain is an open-source framework for building AI pipelines. It was created by Harrison Chase in late 2022, and it has become the most widely used framework for connecting language models to the real world.

There are four core elements you need to understand.

First, LLMs. LangChain gives you a unified interface to every major model -- GPT, Gemini, Claude. Same code, swap models in one line. You do not rewrite your application when you change providers.

Second, Prompt Templates. Instead of hardcoding prompts, you create templates with variables. This makes your prompts reusable at scale. One template, a thousand different inputs.

Third, Memory. By default, LLMs are stateless -- they forget everything between calls. LangChain adds memory so your application can maintain conversation history across multiple interactions.

Fourth, Chains. This is the core idea. A chain connects steps together. The output of step one becomes the input of step two. Summarize, then translate, then format -- three steps, one chain.

The modern way to build chains is called LCEL -- LangChain Expression Language. It uses the pipe operator to connect components. It is clean, it is readable, and it is powerful.

Let me show you a quick preview of what that looks like in code.

Let me walk you through building your first chain with LCEL.

The syntax is beautifully simple. You write: prompt, pipe, llm, pipe, output_parser. That is it. Three components connected with the pipe operator.

Here is what happens. The user asks a question. The prompt template wraps it with structure and context. The model generates a response. The output parser extracts the clean text.

Now let us extend it. Add memory -- and suddenly the model remembers what the user said before. Add a second step -- and now the output of the first model becomes the input of the next.

In three lines of code, you have a pipeline that would take dozens of lines without LangChain. And the best part? Every component is swappable. Change the model, change the prompt, change the parser -- the pipeline still works.

That is the power of composability. And that is what you will build in Lab 1.

Now let us take LangChain and apply it to something real -- the Beer Game.

The Beer Game is a supply chain simulation invented at MIT in the 1960s. It has been used in business schools around the world to teach one of the most important concepts in supply chain management: the bullwhip effect.

Here is the idea. You have four roles in a supply chain: Retailer, Wholesaler, Distributor, and Factory. Each one orders inventory from the next. The problem? There are delays. And when demand changes at the retail level, each player overreacts. Small changes in demand amplify into massive swings at the factory level.

Now here is where it gets interesting. What if each supply chain role is an AI agent? Each agent has its own prompt, its own inventory data, its own decision-making logic. And we chain them together using LangChain.

The Retailer agent sees customer demand and places an order. That order flows to the Wholesaler agent. The Wholesaler sees the order and makes its own decision. And so on, up the chain.

When you run this simulation over 20 rounds, you will see the bullwhip effect emerge naturally -- from the AI agents' decisions. This is not a toy example. This is how companies model supply chain decisions.

This week you went from single API calls to multi-step pipelines.

You built chains that connect prompts, models, and memory into coherent workflows. And you put them to the test with the Beer Game -- watching AI agents make supply chain decisions and the bullwhip effect emerge from their interactions.

That is the power of orchestration. Not one model answering questions -- multiple steps working together toward a goal.

Why Chains? The Single-Call Limitation

A single API call is powerful. But real applications need multiple steps working together.

The Problem with Single Calls

Until now, every interaction with an LLM has been a single round trip. You send a prompt, you get a response. That works for simple tasks -- answering a question, summarizing a paragraph, generating a code snippet.

But what happens when the task is more complex? What if you need to:

  • Summarize a document, then translate the summary, then format it as an email
  • Look up data from a database before the model can answer
  • Remember what the user said three messages ago
  • Call a tool, process the result, and pass it to a second model

A single API call cannot do any of this. You need a pipeline.

What You Need: An Orchestration Framework

You could build all of this from scratch -- manually passing outputs between functions, managing memory yourself, handling errors at every step. But that is tedious and error-prone.

That is why frameworks like LangChain exist. They give you pre-built, tested components for every part of the pipeline -- and a clean way to connect them together.

The Four Building Blocks

Every LangChain application is built from four fundamental components.

🤖

LLMs

Connect to GPT, Gemini, Claude -- same code, swap models in one line. LangChain provides a unified interface so your pipeline works regardless of the provider.

📝

Prompt Templates

Variables + structure = reusable prompts at scale. One template, a thousand different inputs. No more copy-pasting and editing prompts manually.

🧠

Memory

Conversation history that persists across calls. LLMs are stateless by default -- Memory lets your application remember what the user said before.

🔗

Chains

Connect steps together. Output of step 1 becomes input of step 2. This is the core abstraction -- composable, reusable, testable pipelines.

LCEL: The Modern Syntax

LangChain Expression Language (LCEL) uses the pipe operator to connect components. It is clean, declarative, and makes your pipelines readable at a glance.

# LCEL syntax -- the pipe operator connects components chain = prompt | llm | output_parser # Invoke the chain result = chain.invoke({"topic": "supply chain optimization"})

The Beer Game: A Supply Chain Classic

A simulation game invented at MIT in the 1960s that teaches one of the most important lessons in supply chain management.

What Is the Beer Game?

The Beer Distribution Game was developed at MIT Sloan School of Management by Jay Forrester in the 1960s. It simulates a simple supply chain with four roles: Retailer, Wholesaler, Distributor, and Factory. Each player manages inventory and places orders to the player upstream.

The game teaches a powerful lesson: even with stable customer demand, supply chains naturally amplify variability. This is the bullwhip effect.

The Bullwhip Effect

Small changes in customer demand at the retail level create increasingly large swings as you move upstream. The retailer overorders slightly. The wholesaler overorders more. By the time the signal reaches the factory, a 5% increase in customer demand can look like a 40% surge.

Why It Matters

The Beer Game teaches systems thinking -- how delays, incomplete information, and local optimization create global problems. These are the same dynamics that cause real supply chain failures, from semiconductor shortages to the toilet paper crisis of 2020.

LangChain in Industry

How companies use LangChain -- and when you should (and should not) reach for it.

Where LangChain Is Used

  • Customer Support Systems: Multi-step pipelines that retrieve context, classify intent, and generate personalized responses
  • Document Analysis: Chains that extract, summarize, and cross-reference information from large document sets
  • Research Pipelines: Automated workflows that search, synthesize, and format findings from multiple sources
  • Internal Tools: Enterprise chatbots that connect to databases, APIs, and internal knowledge bases
