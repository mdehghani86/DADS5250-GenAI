Video 2: "How RAG Works"

Voiceover + animation -- 7-8 minutes -- 4 segments
SEG A The Hallucination Problem
Voiceover + Animation ~2 min

Before we fix the problem, let us see it clearly.

Ask GPT about your company's Q4 revenue. It does not know your company. But it will not say "I do not know." It will fabricate numbers. Real-looking numbers -- with dollar signs, growth percentages, quarter-over-quarter comparisons. All fake.

This is hallucination. The model generates text that sounds right but is factually wrong. It is not lying -- it is predicting the most likely next words. And sometimes those words form statements that are completely false.

In a classroom, that is embarrassing. In production -- legal contracts, medical advice, financial reports -- it is dangerous.

Storyboard -- Hallucination vs Grounded

SEG B The RAG Pipeline Overview
Voiceover + Animation ~2 min

RAG -- Retrieval-Augmented Generation -- solves this with one simple idea: before the model generates an answer, give it the real data first.

There are three core steps. First, you chunk your documents -- break them into smaller pieces that each contain one idea. Second, you embed those chunks -- convert them into numerical vectors that capture their meaning. Third, when a user asks a question, you retrieve the most relevant chunks and feed them to the LLM as context.

The model still generates the response. But now it has real data to work with. It is the difference between a closed-book exam and an open-book exam.

Storyboard -- The RAG Pipeline (3 Steps)
MEDIA TEAM: This is the signature animation of M05. Left-to-right flow. Documents on the left fracture into chunks. Chunks transform into glowing particles that settle into a 3D space. Query arrow highlights nearest particles. Particles feed into LLM node. Answer emerges on the right.
SEG C What Are Embeddings
Voiceover + Animation ~2 min

Let me explain embeddings, because this is the key insight.

An embedding is a list of numbers -- a vector -- that represents the meaning of a piece of text. Here is what matters: texts with similar meanings have similar vectors. "How do I return a product?" and "What is the refund policy?" have completely different words. But their embeddings are nearly identical -- because they mean nearly the same thing.

This is what makes RAG possible. When a user asks a question, you convert the question into a vector. Then you find the stored vectors that are closest to it. Those are your most relevant chunks. No keyword matching. No exact string search. You are searching by meaning.

Storyboard -- Meaning Has Geometry
MEDIA TEAM: Dark background, glowing particles. Color-code clusters. When the search query enters, show a golden beam connecting to the nearest dots. This visualization should feel like "meaning has geometry."
SEG D Vector Stores
Voiceover + Animation ~1 min

You store these embeddings in a vector database -- a database optimized for similarity search. Instead of "find me the row where name equals X," you say "find me the five vectors closest to this query."

We will use ChromaDB in the labs. It is open source, runs locally, and does not need a server. In production, you would look at Pinecone or FAISS for scale. But the concept is the same everywhere.

Video 3: "Building a RAG Pipeline"

Voiceover + code display -- 6-7 minutes -- 3 segments
SEG A The 5-Step Build
Voiceover + Animation ~2 min

Let us build this thing. Five steps. That is the entire pattern.

Step one: load your documents. PDFs, text files, web pages -- LangChain has loaders for all of them. Step two: split them into chunks. We use the RecursiveCharacterTextSplitter -- it tries paragraph breaks first, then sentences, then characters. Smart splitting. Step three: create embeddings. One API call converts each chunk into a 1536-dimensional vector.

Step four: store those embeddings in ChromaDB. Step five: query. User asks a question, we embed the question, find the closest chunks, and pass everything to the LLM.

Five steps. Load, split, embed, store, query. That is every RAG system you will ever build.

SEG B Chunking Strategies
Voiceover + Animation ~2 min

Chunking deserves its own discussion, because how you split your documents determines the quality of your retrieval.

Too small, and each chunk is a fragment -- no context, no meaning. Too large, and the relevant sentence is buried in three paragraphs of noise. The sweet spot for most use cases is 500 to 1000 characters with 50 to 100 characters of overlap between chunks.

The overlap is important. Without it, an idea that spans a chunk boundary gets lost. With overlap, both chunks capture the boundary text.

There is no universal right answer. A legal contract needs different chunking than a product manual. Experiment.

SEG C The Code
Voiceover + Code ~2 min

Here is what the core pipeline looks like in code.

# 1. Load
from langchain_community.document_loaders import PyPDFLoader
docs = PyPDFLoader("report.pdf").load()

# 2. Split
from langchain.text_splitter import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3-4. Embed + Store
from langchain_openai import OpenAIEmbeddings
from langchain_chroma import Chroma
vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

# 5. Query
results = vectorstore.similarity_search("What was Q4 revenue?")

That is the entire pipeline. Twelve lines. Load, split, embed, store, query. You will build this yourself in Lab 2.

Video 4: "RAG in Production"

Voiceover + animation -- 5-6 minutes -- 3 segments
SEG A Evaluating RAG
Voiceover + Animation ~2 min

You built a RAG pipeline. It runs. It returns answers. But how do you know if it is any good?

There are two things to evaluate. First, retrieval quality -- did the system find the right chunks? If you ask about Q4 revenue and it retrieves the HR policy, no amount of generation will fix that. Second, generation quality -- given the right chunks, did the model use them correctly? Or did it ignore the context and hallucinate anyway?

In production, you test both. You build a set of test questions with known answers, run them through your pipeline, and measure. Is the answer in the retrieved chunks? Does the generated response match the ground truth?

Storyboard -- Two Things to Evaluate
SEG B Common Failures
Voiceover + Animation ~2 min

Let me walk you through the three most common RAG failures.

Number one: bad chunking. Your chunks are too large, and the relevant sentence is buried. Or too small, and the chunk is meaningless on its own. Fix: experiment with chunk size and overlap.

Number two: irrelevant retrieval. The query is about pricing but the system returns product descriptions because the embeddings are too generic. Fix: better embedding models, metadata filtering, or hybrid search combining semantic and keyword search.

Number three: context ignored. The model has the right chunks but generates an answer from its training data anyway. Fix: stronger system prompts -- explicitly tell the model "only answer from the provided context."
