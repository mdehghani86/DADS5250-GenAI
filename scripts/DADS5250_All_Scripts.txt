MODULE M04: LANGCHAIN & BEER GAME
================================================================

Video 2: "LangChain Foundations"

PART A: The Four Building Blocks

LangChain is an open-source framework for building AI pipelines, created by Harrison Chase in late 2022. There are four core elements.

First, LLMs -- a unified interface to every major model. Same code, swap providers in one line. Second, Prompt Templates -- reusable prompts with variables. One template, a thousand inputs.

Third, Memory -- LLMs are stateless by default, so LangChain adds conversation history across calls. Fourth, Chains -- connecting steps together. The output of step one becomes the input of step two.

---

PART B: Workflow Modeling for AI

Here is what LangChain actually does, with an example you already know.

Imagine a process inside an Amazon fulfillment center. You click "buy" on your phone, and that one click kicks off a whole pipeline. A picker walks the aisle and grabs your item from the shelf. A packer drops it in the right-sized box. A labeler prints the shipping label and slaps it on. A sorter routes the box to the truck going to your zip code. And a driver delivers it to your door. Each station has one job. Each station is independent. And the conveyor belt between them is what makes the whole thing work -- it carries the package from one station to the next, without anyone having to carry it by hand.

LangChain does the exact same thing for AI workflows. Each of the four building blocks you just saw -- prompt templates, LLMs, memory, chains -- is a station on your AI fulfillment line. Input comes in one end. Each component does its one job. The finished output comes out the other end.

And here is the beautiful part. Just like Amazon can swap out the labeling machine without shutting down the entire warehouse, you can swap one component in your LangChain pipeline -- change the LLM, update the prompt template, add memory -- without touching the rest of your application. That is what makes LangChain powerful. LangChain is a fulfillment center for AI.

---

PART C: LCEL -- The Pipe Operator

The modern way to build chains is called LCEL -- LangChain Expression Language. It uses the pipe operator to connect components. It is clean, it is readable, and it is powerful.

Here is a quick preview of what that looks like in code.

This is how you build your first chain with LCEL.

The syntax is beautifully simple. You write: prompt, pipe, llm, pipe, output_parser. That is it. Three components connected with the pipe operator.

PART B: Adding Memory and Multi-Step

Now let us extend it. Add memory -- and suddenly the model remembers what the user said before. Add a second step -- and now the output of the first model becomes the input of the next.

In three lines of code, you have a pipeline that would take dozens of lines without LangChain. And the best part? Every component is swappable. Change the model, change the prompt, change the parser -- the pipeline still works.

That is the power of composability. And that is what you will build in Lab 1.

================================================================

Video 1: "The Spark" -- The Support Call

PART 01: The Support Call

Think about what happens when you call customer support. The first thing they ask is -- what is your issue? And depending on your answer, everything changes. Billing goes one direction. Technical goes another. Returns goes somewhere else. And if the first fix does not work, they loop back and try something different.

That is not a pipeline. That is a graph -- with branches, conditions, and loops.

---

PART 02: Key Message -- LangGraph

Most AI systems you have built so far run in a straight line. Input goes in, output comes out. But real tasks do not work that way. They branch. They loop. They depend on what just happened. And the tool that lets you design these kinds of workflows in Python is called LangGraph.

Before we get to LangGraph, though, we need to understand what powers it -- agents and tools. That is where we start.


Video 4: "Introduction to LangGraph"

PART A: Why LangGraph

Remember the customer support call from the opening? Billing goes one way, technical goes another, and if the first fix fails, you loop back and try something different. That was a graph. LangGraph lets you lay out your entire agent workflow exactly like that. You decide the steps, the connections, and the branches. The AI navigates that graph at runtime, choosing its own path based on what it finds.

To build one, you need three pieces. Nodes, edges, and state.

---

PART B: Nodes, Edges, State

A node is a Python function -- it receives state, does one thing, and passes updated state forward. An edge connects nodes. Conditional edges branch: "If the LLM wants a tool, go to the tool node. If done, go to the end."

State is the shared memory. Every node reads from it and writes to it. By the time you reach the end, state contains the full record of everything that happened.

---

PART C: LangGraph in Context

You may ask -- why learn LangGraph when model providers now ship their own agent SDKs? OpenAI has one, Anthropic has one, Google has one. For simple tasks, they work out of the box. But the moment your workflow gets complex -- multiple branches, conditional loops, human approval steps -- those SDKs are not enough. LangGraph gives you full control over every decision point.

Think of it this way. The SDKs are like a ride-sharing app. LangGraph is like having the keys to the car. When the road gets complicated, you want the keys. We cover the newer SDKs in Module 12.

---

PART D: The 5-Step Pattern

Building a LangGraph agent follows five steps. Every time.

Step one: define the state. What information flows through the graph? Messages, tool results, decisions -- all stored in a typed dictionary.

Step two: create nodes. Each node is a Python function that takes state and returns updated state.

Step three: add edges. Connect your nodes. "After classify, go to respond."

Step four: add conditional edges. "After the LLM responds, check: did it want to call a tool? If yes, go to the tool node. If no, go to the end." This is the decision-making.

Step five: compile and run. Call graph.compile() to get a runnable object. Invoke it with your initial state.

---

PART E: The Code

Here is a minimal LangGraph agent.

Look at the conditional edge. That is where the intelligence lives. After the agent responds, we ask one question: did the LLM request a tool call? If yes, we route to the tools node. The tools node runs the function, gets the result, and routes back to the agent. The agent sees the new information, thinks again, and decides whether to call another tool or finish.

That loop can run once, or it can run ten times. The agent keeps going until it decides it has enough to answer. And that is the ReAct pattern you learned earlier, except now it is not a concept. It is a graph you can see, debug, and control.

Video 2: "CrewAI: Building Agent Teams"

PART A: The Story Behind CrewAI

CrewAI was created in late 2023 by Joao Moura, a Brazilian-born engineer who noticed the same problem over and over -- developers were writing hundreds of lines of boilerplate just to get two agents to talk to each other. His idea: what if multi-agent systems worked the way real teams work? Define who is on the team, what each person does, and how they collaborate. The framework handles the rest.

He open-sourced it in early 2024 and it took off. By 2025, CrewAI had raised funding and become one of the most popular multi-agent frameworks in Python. What makes it special is the abstraction. You do not think in nodes and edges. You think in people and jobs.

---

PART B: The Three Building Blocks

Everything in CrewAI comes down to three things. An agent has a role, a goal, and a backstory. A task is the assignment -- a description, an expected output, and one assigned agent.

The crew brings it together. You list your agents and tasks, pick the process -- sequential or hierarchical -- and hit run. CrewAI handles execution, handoffs, and context sharing.

---

PART C: The Code

Here is what it looks like in code.

The researcher runs first, finds the papers. Its output automatically flows to the writer. The writer uses those findings to produce the summary. Two agents, two tasks, one crew. That is multi-agent AI.

---

PART D: Sequential vs Hierarchical

Once you have your agents and tasks, you need to decide how they work together. CrewAI gives you two options.

The first is sequential. The researcher goes first, finishes, and hands everything to the writer. The writer finishes and hands everything to the reviewer. One after another, like a production line. You know exactly what happens and in what order.

The second is hierarchical. Instead of a fixed order, you add a manager agent. The manager reads the overall goal, decides which agent should work next, reviews the output, and can send it back if it is not good enough. It is closer to how a real project team operates.

For most tasks, sequential is the right starting point. You will know when you need hierarchical because sequential will start feeling too rigid.

---

PART E: Delegation

One of CrewAI's most powerful features is delegation. An agent can ask another agent for help.

The writer is drafting a section and realizes it needs a specific statistic. Instead of guessing, it delegates to the researcher: "Find me the 2024 market size for vector databases." The researcher runs, finds the number, and passes it back. The writer continues with real data.

This is not hardcoded. The agents decide when to delegate based on the task at hand. It is collaboration, not just sequencing.

Video 2: Vision APIs -- Sending Images to AI

PART A: How to Send Images

Sending an image to an AI model is simpler than you think.

With OpenAI's multimodal models, you send a message that contains both text and an image. The image can be a URL or a base64-encoded file. The model processes both together and responds.

With Gemini, the approach is similar but the API structure differs slightly. You can send images, PDFs, even short video clips.

---

PART B: What Vision Models Can Do

Vision models can do more than you might expect. Object detection, text extraction from photos, diagram analysis, comparing two images, describing scenes, reading handwriting. The accuracy varies by task, but for many practical applications it is remarkably good.

---

PART C: Conversational Vision

The key insight is this. Vision is not a separate capability. It is integrated into the same conversation. You can ask follow-up questions about an image. You can send multiple images and ask the model to compare them. It is a conversation, not a one-shot analysis.

---

PART D: Vision API Costs

Vision calls cost more than text calls. That is the trade-off. An image adds tokens to your request, and those tokens are not cheap.

With OpenAI, a single image can add anywhere from a few hundred to several thousand tokens depending on the resolution. High-detail mode costs more but gives better results for complex images like engineering diagrams. Low-detail mode is cheaper and works fine for simple photos.

With Gemini, the pricing is more generous. Google's free tier includes vision calls, which makes it a great option for prototyping and student projects. For production workloads, their paid tier is competitive.

The practical lesson is this. Do not send high-resolution images when a thumbnail would do. Resize before you send. Use low-detail mode for classification tasks and high-detail mode only when you need precision. A little optimization goes a long way on your API bill.

---

PART E: When Vision Makes Sense

Not every problem needs vision. If the data is already in text form, sending a screenshot of a spreadsheet to a vision model is a waste of money. Just send the CSV.

Vision makes sense when the information is inherently visual. A photo of a damaged product. An engineering diagram. A handwritten form. A chart someone pasted into a slide deck without sharing the data. Those are the cases where text alone cannot capture what you need, and vision becomes the right tool.

PART B: What Claude Code Can Do

Claude Code reads files, writes code, runs terminal commands, creates and edits files, searches the web, and manages git. It installs packages, executes tests, creates commits, and handles branches.

The key difference from a chatbot: it has access to your codebase. It sees everything. It reasons about your project as a whole. Changing a function signature in one file means updating every caller in three other files, and Claude Code knows that. It knows your test suite expects a specific return type. It traces dependencies across your entire project.

The difference between a coding chatbot and Claude Code is the difference between a spell checker and a co-author.

---

PART C: The Trust Model

Claude Code proposes changes. You approve them. It asks before running commands, before editing files, before pushing code. You stay in control.

Every action goes through a permission step. Claude Code shows you exactly what it plans to do: the file it wants to edit, the command it wants to run, the commit message it wants to use. You approve or reject each one. Nothing happens without your explicit consent.

You can configure trust levels. Allow all file reads but require approval for writes. Allow test execution but block pushes to remote repositories. The default is conservative: ask before everything. You relax the controls as you build confidence.

The best way to use it is to treat it like a junior engineer who is very fast but needs supervision. It proposes. You decide.

Video 2: "Google A2A Protocol"

PART A: What is A2A

A2A stands for Agent-to-Agent. It is an open protocol published by Google in April 2025. The problem it solves is straightforward: agents built by different teams, using different frameworks, running at different companies, have no standard way to talk to each other.

Think about what HTTP did for the web. Before HTTP, every network had its own protocol. After HTTP, any browser could talk to any server. A2A aims to do the same thing for AI agents. Any agent that implements the A2A spec can discover, communicate with, and delegate tasks to any other A2A-compatible agent.

Google did not build this alone. Over 50 technology partners were involved at launch, including Salesforce, SAP, LangChain, and CrewAI. The spec is open. Anyone can implement it.

---

PART B: How A2A Works

A2A has three core concepts. Agent Cards, Tasks, and streaming updates.

An Agent Card is like a business card for an AI agent. It is a JSON document hosted at a well-known URL that describes what the agent can do, what inputs it expects, and what authentication it requires. When another agent wants to find a collaborator, it reads the Agent Card first.

Once an agent finds a partner, it sends a Task. A Task is a structured request with a clear goal. The receiving agent processes it and returns results, either all at once or as a stream of updates. Streaming is important because many agent tasks take time, and the calling agent needs to know what is happening.

The flow is: discover the Agent Card, create a Task, receive updates, get the final result. Four steps. That is the entire protocol.

---

PART C: Why A2A Matters

The real value of A2A is interoperability. Right now, if you build an agent in CrewAI, it only talks to other CrewAI agents. If someone else builds an agent in LangGraph, those two systems cannot collaborate without custom glue code.

A2A changes that. Your CrewAI research agent publishes an Agent Card. Someone else's LangGraph analysis agent reads that card, sends a task, and gets results back. The frameworks do not matter. The protocol handles the translation.

This is how enterprise AI will work. Not one company building every agent, but many teams building specialized agents that discover and collaborate with each other through a shared standard. A2A is that standard.

Video 1: "The Spark"

PART 01: The Hook

Move fast and break things. That was Silicon Valley's motto for over a decade. Every startup founder has said it. But none of them were building systems that write legal contracts, screen resumes, or advise patients.

This module is about building AI that is fast AND careful. You have spent the whole course learning how to make AI powerful. Now you learn how to make it responsible.

---

PART 02: What This Module Covers

We are covering three layers. Prompt injection, where users trick your AI into doing things it should not. Output safety and bias, where your AI leaks data or quietly discriminates. And guardrails, the engineering stack that catches all of it before your users see the damage.

By the end of this module, you will know how to build AI that is both powerful and trustworthy. That is the professional standard.


Video 2: "Prompt Injection &amp; Attacks"

PART A: What Is Prompt Injection

Every LLM application has a system prompt -- the rulebook that tells the model who it is and what it should refuse. Prompt injection is when a user rewrites that rulebook.

A user types: "Ignore all previous instructions." And sometimes, the model obeys -- not because it is malicious, but because it cannot distinguish your instructions from the user's. They are all just text in a context window.

This is not theoretical. Researchers tricked Bing Chat into revealing its full system prompt. Customer service bots were manipulated into unauthorized refunds and leaked pricing data. The models were just being helpful.

---

PART B: Types of Attacks

Three flavors. Direct injection: the user types a malicious instruction straight into the chat. Blunt, obvious, and it still works more often than you would think.

Indirect injection: the attack is hidden inside a document or web page the AI retrieves. The user never typed anything malicious. The poison was in the data.

Jailbreaking: users craft elaborate roleplay scenarios to bypass safety filters. It is a constant arms race between model providers and the people trying to break them.

---

PART C: Defenses

No single defense is bulletproof, but layering them helps. Start with input validation -- scan for known injection patterns before anything reaches the LLM. A regex catches the obvious attacks. A classifier catches the subtle ones.

Harden your system prompt with the sandwich defense -- put critical instructions at both the beginning and end. Separate your channels: user input in the user message field, system instructions in the system message field. Never concatenate them.

And assume the worst. Validate output too. If the bot suddenly responds in a different language or leaks your system prompt, catch it before the user sees it.


Video 6: "Building Guardrails"

PART A: Input Guardrails

Input guardrails are everything between the user pressing Enter and the LLM seeing the request. Think of them as a bouncer at the door -- they stop obvious problems from walking in.

Length limits first. Cap input length and return a clear error. Next, content classification -- run the request through a lightweight classifier before it reaches the LLM. The OpenAI Moderation API handles safety; a custom classifier handles your domain rules.

Finally, structured input validation. If a field should be a dropdown, do not let users free-text into it. Every dropdown instead of a text box is one fewer attack vector.

---

PART B: Output Guardrails

Output guardrails inspect what the LLM produces before the user sees it. Force the model to return JSON matching a strict Pydantic schema. If a confidence field must be between 0 and 1 and the model returns 1.5, validation rejects it automatically.

For factual accuracy, add a verification step -- a second LLM call or database lookup checks claims against known sources. For high-stakes applications, that latency is far cheaper than a lawsuit.

Confidence scoring is the last piece. If the model's confidence is below your threshold, show it as a suggestion or escalate to a human. Low confidence is not a failure. It is a signal.

---

PART C: System Guardrails

System guardrails protect the infrastructure. Rate limiting prevents one user from burning your API budget. Cost caps set a hard ceiling so a runaway loop does not generate a five-figure invoice overnight.

Logging is non-negotiable. Every request, response, and guardrail trigger -- logged with a timestamp, user ID, and reason. When something goes wrong, you need to reconstruct exactly what happened.

Monitoring ties it together. A spike in blocked requests might mean an attack. A drop in confidence scores might mean your data changed. Your guardrails generate signals. Monitoring turns those signals into action.
