Video 2: "What Are LLMs?"

PART A: What Is a Large Language Model

LLM stands for Large Language Model. Here is what that means.

Take a significant portion of human knowledge -- books, code, research, conversations. Compress it into a mathematical model. The model does not store the text word-for-word. It learns patterns.

When you give it a prompt, it uses those patterns to generate something new. Every response is original. Every piece of code is composed, not retrieved. That is generative AI -- and LLMs are the engine behind it.

---

PART B: Platform vs API

You can access these models two ways.

The first is through a platform -- ChatGPT, Gemini, Claude. You open a browser, type a message, get a response. Think of it as walking into a restaurant and ordering at the counter. It works. It is convenient.

The second way is through an API. Your Python code sends a request directly to the model. Think of this as building the delivery app -- you handle thousands of orders, route them, process them, all automatically. Same kitchen, same food, completely different power.

---

PART C: How LLMs Generate Text

Here is something worth understanding. LLMs do not think. They predict.

When you send a prompt, the model looks at all the words so far and asks one question: what is the most likely next word? It picks one, adds it, and repeats. Word by word. Token by token.

This is why temperature matters. At zero, it always picks the safest word. At higher values, it takes risks. And this is why LLMs sometimes hallucinate. They generate text that sounds right but is factually wrong.

The model does not know what is true. It knows what is likely.


Video 3: "What Is an API?"

PART A: API from Zero

API stands for Application Programming Interface. Let me make that simple.

Think of a vending machine. You put something in -- a coin and a button press. Something happens inside that you cannot see. And something comes out -- your drink.

You do not need to know how the machine works inside. You just need to know two things: what to put in, and what comes out.

An LLM API works the exact same way. You put in a prompt -- your question. Something happens on the model provider's servers -- the model thinks. And out comes a response -- the answer.

And here is the important part: every major LLM has an API. OpenAI has one. Google has one for Gemini. Anthropic has one for Claude. Meta's Llama, Mistral, xAI's Grok -- all of them. Different companies, different models, but the same idea. Once you know how to use one API, you can use them all. The pattern is the same: send a prompt, get a response.

Video 5: "Parameters, Tokens &amp; Temperature"

PART A: Three Things You Control

Before you start Lab 2, let us talk about what you are actually controlling when you call the API.

Three things matter: temperature, tokens, and cost. Let us take them one at a time.

---

PART B: Temperature

First: temperature. Temperature controls randomness.

At zero, the model gives you the same answer every time -- the most predictable, most likely response. Ask it the same question twice, you get the same reply twice. This is what you want for tasks like data extraction, classification, or anything where you need repeatable output.

At one, the model takes more risks. Some creativity, some variation. This is the sweet spot for most conversational tasks.

At two, the model gets very creative -- sometimes too creative. Useful for brainstorming, poetry, or ideation. Less useful when you need accuracy.

Rule of thumb: if you want reliability, go lower. If you want variety, go higher.

---

PART C: Tokens

Second: tokens. Tokens are how LLMs measure text.

A token is roughly four characters, or about three-quarters of a word. The word "building" might be one token, or it might be split into "build" and "ing" -- depending on the tokenizer. Short common words are usually one token. Long or rare words often split into two or three.

Here is why this matters. When you send a prompt, you pay for input tokens. When the model responds, you pay for output tokens. Both counts show up on your bill. The model also has a maximum context window -- the total number of tokens it can handle in one call. Go over, and the request fails.

So tokens are two things at once: the unit of measurement, and the unit of billing.

---

PART D: Cost

Third: cost. Here is the good news. This technology is remarkably cheap.

GPT-4.1 Mini, the model you will use for most of this course, costs about fifteen cents per million input tokens. A million tokens is roughly seven hundred and fifty thousand words -- more than ten full-length novels. For fifteen cents. Output tokens are a bit more expensive, but the total cost for most labs in this course will land around a few cents per run.

That changes how you think about building. You do not have to be stingy. You can run a prompt a thousand times while testing. You can let your script iterate. Cost is rarely the limiting factor anymore -- time and design are.

You will experiment with all three of these -- temperature, tokens, and cost -- in Lab 2.

Video 3: "Prompting Strategies"

PART A: Intro — Why Strategies Matter

Now that you know the anatomy of a prompt, let me tell you something that might surprise you. Large language models actually enjoy getting examples. Just like us.

Think about it. If I asked you to "classify these customer reviews" with no context, you would probably ask me to show you a few first. "What does a positive one look like? What does a tricky one look like?" You would want to see the pattern before you start.

LLMs work the same way. Sometimes they can figure it out on their own. Sometimes they need a few examples. And sometimes, for harder problems, they need to slow down and think out loud. These three modes have names: zero-shot, few-shot, and chain-of-thought. In this video, we will walk through all three -- when to use each one, and why.

---

PART B: Zero-Shot Prompting

Let us start with zero-shot. Zero-shot means exactly what it sounds like -- you give the model a task with zero examples. You just tell it what to do and trust it to figure out the pattern.

Here is a classic zero-shot prompt: "Classify this review as positive, negative, or neutral." No examples. No training. Just the task. And most of the time, this works beautifully -- the model has seen millions of reviews during training, so it knows what positive and negative look like.

But what about edge cases? "The food was amazing but the service ruined everything." Is that positive or negative? A human would probably say "mixed." The model might say "positive" one time and "negative" the next, depending on which part it focused on. Same prompt, different answer. That is the risk with zero-shot.

---

PART C: Few-Shot Prompting

Strategy two: few-shot. You give the model a few examples before the actual task. "Here are three reviews with their classifications. Now classify this one." Suddenly the edge case gets consistent results. The model learns the pattern from your examples.

---

PART D: Chain-of-Thought -- The Bat and Ball

Strategy three: chain-of-thought. You tell the model to think step by step. Here is the classic example. "A bat and a ball cost $1.10 together. The bat costs $1.00 more than the ball. How much does the ball cost?"

Without chain-of-thought, most models say 10 cents. That is wrong. With chain-of-thought -- "Think step by step" -- the model works through the algebra and gets 5 cents. Just four words can dramatically improve accuracy on reasoning tasks.

When do you use which? Zero-shot for simple tasks. Few-shot when you need consistent formatting. Chain-of-thought when the task requires reasoning. You will practice all three in Lab 1.


Video 4: "From Text to Structure"

PART A: Intro — The Second Level of Control

So far in this module, you have learned how to talk to the model. You know how to set the role, the task, the context, the format. You know when to use examples and when to ask for step-by-step reasoning. That is the first level of control -- input control.

But here is the thing. Even when you ask the model beautifully, you still get back text. Free-form paragraphs. A block of writing. And that is fine when the answer is for a human to read. But the moment you want to do something with the output -- store it, filter it, feed it into another system -- text becomes a problem.

This video is about the second level of control. Not just what the model says, but the exact shape of what it hands you back. We will cover two tools: JSON mode, which forces the model to return valid JSON, and Pydantic, which makes sure that JSON has exactly the fields and types your code needs.

---

PART B: The Problem with Text

Let me make the problem concrete. The output of a model, by default, is text. Free-form, unstructured text. And text is great for humans -- but terrible for code.

You cannot put a paragraph into a database column. You cannot feed prose into an analytics pipeline. You cannot reliably pull three numbers out of a sentence using string matching without it breaking the moment the wording changes. If your goal is to use the model's output in code, text is the enemy.

What you need instead is structure. You need to know, in advance, that the field called "sentiment" will always be there, and it will always be one of three values, and "confidence" will always be a number between zero and one. That is what JSON gives you -- a contract.

---

PART C: JSON Mode

JSON mode forces the model to return valid JSON -- guaranteed parseable, guaranteed structured.

Watch the difference. Same question, two modes.

Text mode: "The sentiment of the review is positive. The customer expressed satisfaction with the product quality and delivery speed."

JSON mode: {"sentiment": "positive", "confidence": 0.92, "keywords": ["quality", "delivery speed"]}

Same information. But the JSON version can be parsed by code in one line. No regex. No string manipulation. Just json.loads() and you have a Python dictionary.

---

PART D: Pydantic Validation

But JSON mode has a gap. It guarantees valid JSON, but it does not guarantee the right fields or the right types. That is where Pydantic comes in.

Pydantic lets you define a schema -- a class that says "I expect a field called sentiment of type string, a field called confidence of type float between 0 and 1." If the model returns something that does not match, Pydantic catches it immediately. No silent bugs. No bad data reaching your database.


Video 6: "Function Calling -- From Text to Action"

PART A: What Is Function Calling

Until now, the model only produces text -- or structured text. Function calling lets it produce actions.

You define tools -- functions with names, descriptions, and parameter schemas. The model reads the user request and decides: which tool should I use? What arguments should I pass?

The model does not execute the function. It generates a structured call -- a JSON object saying "call this function with these arguments." Your code handles execution.

This is the bridge between a chatbot and an agent. The model reasons about which tool to use. Your code does the work.

---

PART B: The Five-Step Weather Example

Here is a complete example.

Step 1: You define a function called get_weather with parameters city and units.

Step 2: User says "What is the weather in Boston?"

Step 3: The model does not answer directly. Instead it returns: {"name": "get_weather", "arguments": {"city": "Boston", "units": "fahrenheit"}}

Step 4: Your code calls the real weather API with those arguments.

Step 5: You send the result back to the model. Now it generates a human-readable response: "It is currently 72 degrees in Boston with partly cloudy skies."

Five steps. The model decided which function to call. Your code did the work. That is function calling.

Video 2: "LangChain Foundations"

PART A: The Four Building Blocks

LangChain is an open-source framework for building AI pipelines. It was created by Harrison Chase in late 2022, and it has become one of the most widely used frameworks for connecting language models to the real world.

There are four core elements you need to understand.

First, LLMs. LangChain gives you a unified interface to every major model -- GPT, Gemini, Claude. Same code, swap models in one line. You do not rewrite your application when you change providers.

Second, Prompt Templates. Instead of hardcoding prompts, you create templates with variables. This makes your prompts reusable at scale. One template, a thousand different inputs.

Third, Memory. By default, LLMs are stateless -- they forget everything between calls. LangChain adds memory so your application can maintain conversation history across multiple interactions.

Fourth, Chains. This is the core idea. A chain connects steps together. The output of step one becomes the input of step two. Summarize, then translate, then format -- three steps, one chain.

---

PART B: Workflow Modeling for AI

Here is what LangChain actually does, with an example you already know.

Imagine a process inside an Amazon fulfillment center. You click "buy" on your phone, and that one click kicks off a whole pipeline. A picker walks the aisle and grabs your item from the shelf. A packer drops it in the right-sized box. A labeler prints the shipping label and slaps it on. A sorter routes the box to the truck going to your zip code. And a driver delivers it to your door. Each station has one job. Each station is independent. And the conveyor belt between them is what makes the whole thing work -- it carries the package from one station to the next, without anyone having to carry it by hand.

LangChain does the exact same thing for AI workflows. Each of the four building blocks you just saw -- prompt templates, LLMs, memory, chains -- is a station on your AI fulfillment line. Input comes in one end. Each component does its one job. The finished output comes out the other end.

And here is the beautiful part. Just like Amazon can swap out the labeling machine without shutting down the entire warehouse, you can swap one component in your LangChain pipeline -- change the LLM, update the prompt template, add memory -- without touching the rest of your application. That is what makes LangChain powerful. LangChain is a fulfillment center for AI.

Video 1: "The Spark" -- The Context Flip

PART 01: The Context Flip

Let me ask you something. Who was the first person to walk on the moon? Easy -- Neil Armstrong, 1969. Now let me ask you another one. Who was the third engineer hired at your company, and what project did they ship first?

You have no idea. Not because you are not smart -- because you do not have the information.

---

PART 02: Introducing RAG

That is what RAG does. Retrieval-Augmented Generation. Before the AI answers your question, it searches your documents first. It finds the relevant pages. And then it generates an answer grounded in real data.

Closed book to open book. Guessing to knowing. That is the entire idea.

---

PART 03: What You Will Build

This week, you are going to build that open book. You will take a real document, break it into pieces, turn those pieces into numbers a computer can search, store them in a database, and then ask questions that get answered with real data instead of hallucinations.

Let us get started.


Video 2: "How RAG Works"

PART A: The Hallucination Problem

There is a word you are going to hear a lot in AI -- hallucination. Let me show you what it looks like.

Ask an LLM to summarize your company's employee handbook. It has never seen your handbook. But it will not say "I do not know." It will write you a summary anyway -- with policy names, section numbers, even direct quotes. All fabricated. Every word of it. The confidence is impressive. The accuracy is not.

This is hallucination. The model generates text that sounds right but is factually wrong. It is not lying -- it is predicting the most likely next words. And sometimes those words form statements that are completely false.

If that happens on a homework assignment, you might get a bad grade. If it happens in a legal contract or a medical report, someone gets hurt.

---

PART B: The RAG Pipeline Overview

RAG -- Retrieval-Augmented Generation -- solves this with one simple idea: before the model generates an answer, give it the real data first.

There are three core steps. First, you chunk your documents -- break them into smaller pieces that each contain one idea. Second, you embed those chunks -- convert them into numerical vectors that capture their meaning. Third, when a user asks a question, you retrieve the most relevant chunks and feed them to the LLM as context.

The model still generates the response. But now it has real data to work with. It is the difference between a closed-book exam and an open-book exam.

---

PART C: What Are Embeddings

Let me explain embeddings, because this is the key insight.

An embedding is a list of numbers -- a vector -- that represents the meaning of a piece of text. Here is what matters: texts with similar meanings have similar vectors. "How do I return a product?" and "What is the refund policy?" have completely different words. But their embeddings are nearly identical -- because they mean nearly the same thing.

This is what makes RAG possible. When a user asks a question, you convert the question into a vector. Then you find the stored vectors that are closest to it. Those are your most relevant chunks. No keyword matching. No exact string search. You are searching by meaning.

---

PART D: Vector Stores

You store these embeddings in a vector database -- a database optimized for similarity search. Instead of "find me the row where name equals X," you say "find me the five vectors closest to this query."

We will use ChromaDB in the labs. It is open source, runs locally, and does not need a server. In production, you would look at Pinecone or FAISS for scale. But the concept is the same everywhere.

Video 1: "The Spark" -- The Support Call

PART 01: The Support Call

Think about what happens when you call customer support. The first thing they ask is -- what is your issue? And depending on your answer, everything changes. Billing goes one direction. Technical goes another. Returns goes somewhere else. And if the first fix does not work, they loop back and try something different.

PART 02: Key Message -- LangGraph

Most AI systems you have built so far run in a straight line. Input goes in, output comes out. But real tasks do not work that way. They branch. They loop. They depend on what just happened. And the tool that lets you design these kinds of workflows in Python is called LangGraph.

Before we get to LangGraph, though, we need to understand what powers it -- agents and tools. That is where we start.


Video 2: "What Is an Agent?"

PART A: Chains vs Agents

Before we build with LangGraph, it is worth understanding what makes it different from everything you have built so far.

Until now, you have built chains. Step one feeds into step two feeds into step three. Same path every time. That is powerful -- and for many tasks, it is all you need.

But some tasks are open-ended. Research a topic. Debug a problem. Plan a trip. You do not know the steps in advance. You need something that can look at the situation, decide what to do next, and change direction if the first attempt does not work. That is an agent.

---

PART B: Three Ingredients of an Agent

So what actually makes something an agent? Three ingredients. An LLM that can reason. Tools it can reach for. And a loop that lets it keep going until the job is done.

At each step, the agent looks at the current situation, decides what information it needs, picks a tool to get it, reads the result, and then decides what to do next. It is not following a script -- it is making decisions.

---

PART C: The ReAct Loop

Under the hood, every agent runs on a pattern called ReAct -- Reason, Act, Observe. Here is what that actually looks like.

Say a user asks: "What is the weather in Boston, and should I bring an umbrella?" The agent looks at the question and thinks -- I do not have weather data, but I have a weather tool. So it calls the weather API. The result comes back: 62 degrees, 40 percent chance of rain. Now the agent has what it needs. It puts the pieces together -- 40 percent rain, probably bring an umbrella -- and responds.

That whole thing was one loop. Reason about what is missing, go get it, look at what came back, and decide if you are done. If you are not done, go again. That is ReAct. Simple idea, but it is the engine behind every agent you will build in this course.

---

PART D: When to Use Agents

Not every task needs an agent. If the workflow is predictable and the steps are always the same, a chain is the right choice. Summarize a document. Translate text. Extract data from a form. Same path every time.

Agents are for open-ended tasks. Research a topic where you do not know how many searches it will take. Debug a problem where you do not know which files to check. Answer a question that might need multiple tools in an order you cannot predict. The agent figures out the path at runtime.

Chains are efficient and predictable. Agents are flexible and smart. You want both in your toolkit.


Video 4: "Introduction to LangGraph"

PART A: Why LangGraph

Remember the customer support call from the opening? Billing goes one way, technical goes another, and if the first fix fails, you loop back and try something different. That was a graph. And now you have a way to build it.

LangGraph lets you lay out your entire agent workflow as a graph. You decide what the steps are, how they connect, and where the branches happen. The AI navigates that graph at runtime, choosing its own path based on what it finds along the way.

To build one, you need to understand three pieces. Nodes, edges, and state. That is what we cover next.

---

PART B: Nodes, Edges, State

LangGraph has three pieces, and once you see them, the whole thing clicks.

A node is just a Python function. It receives the current state of the conversation, does one thing with it, and passes the updated state forward. That one thing could be calling the LLM, running a tool, or making a routing decision.

An edge connects one node to the next. Some edges are simple. "After the LLM responds, always go to the tool node." But the powerful ones are conditional. "Look at what the LLM just said. If it wants to call a tool, go to the tool node. If it is done, go to the end." That is where the branching happens.

And state is the shared memory that ties it all together. Every node reads from it, every node writes to it. The LLM node adds its response. The tool node adds its result. By the time you reach the end, state contains the full story of everything that happened.

Nodes, edges, and state. That is the whole framework. But you might be wondering why we need it at all.

---

PART C: LangGraph in Context

You may ask -- why learn LangGraph when model providers now ship their own agent SDKs? OpenAI has one. Anthropic has one. Google has one. And they are good. For simple agent tasks, they work out of the box.

But the moment your workflow gets complex -- multiple branches, conditional loops, different models at different stages, human approval steps -- those SDKs are not enough. LangGraph gives you full control over every decision point in your workflow. You see the graph. You own the graph. Nothing is hidden behind an abstraction you cannot inspect.

Think of it this way. The SDKs are like using a ride-sharing app. LangGraph is like having the keys to the car. Both get you there. But when the road gets complicated, you want the keys. We will cover the newer SDKs in Module 12.

---

PART D: The 5-Step Pattern

Building a LangGraph agent follows five steps. Every time.

Step one: define the state. What information flows through the graph? Messages, tool results, decisions -- all stored in a typed dictionary.

Step two: create nodes. Each node is a Python function that takes state and returns updated state.

Step three: add edges. Connect your nodes. "After classify, go to respond."

Step four: add conditional edges. "After the LLM responds, check: did it want to call a tool? If yes, go to the tool node. If no, go to the end." This is the decision-making.

Step five: compile and run. Call graph.compile() to get a runnable object. Invoke it with your initial state.

---

PART E: The Code

Here is a minimal LangGraph agent.

Look at the conditional edge. That is where the intelligence lives. After the agent responds, we ask one question: did the LLM request a tool call? If yes, we route to the tools node. The tools node runs the function, gets the result, and routes back to the agent. The agent sees the new information, thinks again, and decides whether to call another tool or finish.

That loop can run once, or it can run ten times. The agent keeps going until it decides it has enough to answer. And that is the ReAct pattern you learned earlier, except now it is not a concept. It is a graph you can see, debug, and control.

MODULE M08: MULTI-AGENT SYSTEMS (CREWAI)
================================================================

Video 1: "The Spark" -- Why Teams Beat Solo

PART 01: The Hiring Analogy

Imagine you are starting a company. Day one, it is just you. You are the CEO, the developer, the marketer, the accountant, the customer support team, and the one who makes the coffee. You can do all of it. But you cannot do all of it well. Not at the same time.

So what do you do? You hire specialists. A developer who only writes code. A marketer who only thinks about growth. An accountant who only handles the books. Each person focuses on one job. And suddenly, the whole operation gets better.

That is exactly what happens with AI agents.

---

PART 02: Single vs Team

That is the idea behind CrewAI. You define the specialists -- a researcher, a writer, an editor. You give each one a clear role and a clear job. And then you let them collaborate, pass context to each other, and check each other's work.

You do not manage the handoffs. You do not babysit the workflow. You describe the team, and CrewAI runs it. That is what this module is about.


Video 2: "CrewAI: Building Agent Teams"

PART A: The Story Behind CrewAI

CrewAI was created in late 2023 by Joao Moura, a Brazilian-born engineer who had spent years building AI systems at large companies. He noticed the same problem over and over -- developers were writing hundreds of lines of boilerplate just to get two agents to talk to each other.

His idea was simple. What if multi-agent systems worked the way real teams work? You define who is on the team, what each person is responsible for, and how they collaborate. The framework handles the rest.

He open-sourced it in early 2024, and it took off. Within months, CrewAI had tens of thousands of stars on GitHub and a growing community of developers building everything from research assistants to automated content pipelines. By 2025, CrewAI had raised funding and become one of the most popular multi-agent frameworks in the Python ecosystem.

What makes CrewAI special is the level of abstraction. You do not think in nodes and edges. You think in people and jobs. You describe the team, and CrewAI figures out how to run it.

---

PART B: The Three Building Blocks

Everything in CrewAI comes down to three things. An agent, a task, and a crew.

Start with the agent. Imagine you are hiring someone. You would tell them their job title -- "Senior Researcher." You would explain what success looks like -- "Find the three most relevant papers on this topic." And you would give them some background so they know how to approach the work -- maybe they are an expert in academic literature, maybe they specialize in industry reports. That is exactly what you do in CrewAI. You define a role, a goal, and a backstory. And if the agent needs to go out and do something -- search the web, read a file, call an API -- you hand it the right tools.

Next comes the task. A task is the actual assignment. You write a clear description of what needs to happen, you describe what the finished output should look like -- a summary, a report, a list of recommendations -- and you assign it to one of your agents. One agent, one task, one clear deliverable.

Finally, the crew. The crew is where it all comes together. You take your agents, you take your tasks, you pick how they should work -- one after another in sequence, or with a manager coordinating from above -- and you hit run. CrewAI handles the execution, the handoffs, and the context sharing between agents. You designed the team. CrewAI runs it.

---

PART C: The Code

Here is what it looks like in code.

The researcher runs first, finds the papers. Its output automatically flows to the writer. The writer uses those findings to produce the summary. Two agents, two tasks, one crew. That is multi-agent AI.

---

PART D: Sequential vs Hierarchical

Once you have your agents and tasks, you need to decide how they work together. CrewAI gives you two options.

The first is sequential. The researcher goes first, finishes, and hands everything to the writer. The writer finishes and hands everything to the reviewer. One after another, like a production line. You know exactly what happens and in what order.

The second is hierarchical. Instead of a fixed order, you add a manager agent. The manager reads the overall goal, decides which agent should work next, reviews the output, and can send it back if it is not good enough. It is closer to how a real project team operates.

For most tasks, sequential is the right starting point. You will know when you need hierarchical because sequential will start feeling too rigid.

---

PART E: Delegation

One of CrewAI's most powerful features is delegation. An agent can ask another agent for help.

The writer is drafting a section and realizes it needs a specific statistic. Instead of guessing, it delegates to the researcher: "Find me the 2024 market size for vector databases." The researcher runs, finds the number, and passes it back. The writer continues with real data.

This is not hardcoded. The agents decide when to delegate based on the task at hand. It is collaboration, not just sequencing.


Video 3: "Orchestrating Complex Workflows"

PART A: Task Dependencies

Real workflows have dependencies. The analyst cannot analyze until the researcher has gathered data. The writer cannot write until the analyst has produced insights. The editor cannot edit until the writer has a draft.

CrewAI handles this through task ordering and context passing. Each task's output becomes available to downstream tasks. You define the order, and the framework ensures that each agent gets the context it needs from the agents that ran before it.

---

PART B: Memory and Context

CrewAI also handles memory. When the researcher finishes, its output is automatically passed to the next agent as context. The analyst does not just see its own task description -- it sees everything the researcher found. And when the analyst finishes, the writer sees both the research and the analysis.

Think of it like a relay race. Each runner passes the baton to the next. The baton is the accumulated context -- and it gets richer at every handoff. You do not wire this up manually. You define the task order, and CrewAI handles the rest.

---

PART C: Real-World Patterns

CrewAI is already running in production at companies like Uber, LinkedIn, and hundreds of others. The use cases follow the same pattern every time.

A content team where one agent researches, another writes, and a third edits. A lead scoring pipeline where one agent qualifies the prospect, another checks the CRM, and a human reviewer approves or rejects. An email responder where one agent reads the incoming message, classifies the intent, and routes it to a specialist agent that drafts the reply.

The pattern is always the same. Break the job into roles. Give each role one clear task. Let CrewAI handle the handoffs.


Video 5: "CrewAI Tools &amp; Processes"

PART A: CrewAI Tools &mdash; Giving Agents Superpowers

In the last video you saw how to define agents, assign them roles, and wire them into a crew. But there is a piece we have not talked about yet -- and it changes everything.

Tools. A CrewAI tool is a Python function that an agent can call during its work. Search the web. Read a file. Query a database. Scrape a website. Send an email. Calculate a number. Without tools, an agent can only think. With tools, it can act.

CrewAI ships with a library of built-in tools -- SerperDevTool for web search, ScrapeWebsiteTool for pulling web content, FileReadTool for reading local files, and more. But the real power is custom tools. You write a Python function, decorate it with @tool, give it a good docstring, and the agent knows how to use it.

Here is the key insight: the quality of your tools determines the quality of your crew. An agent with bad tools is like a surgeon with dull instruments -- smart, but limited.

---

PART B: Processes &mdash; Sequential vs Hierarchical

You already know the difference between sequential and hierarchical from the last video. Now let us talk about when you actually switch from one to the other.

Sequential breaks down when the task requires back-and-forth. If your editor finds a factual error, in sequential mode the crew is already done. There is no way to send it back to the researcher. In hierarchical mode, the manager catches that and says "researcher, verify this claim before we continue."

Hierarchical also shines when you do not know the right order in advance. Maybe some queries need research first, others need analysis first. The manager figures that out at runtime based on what the user asked.

The trade-off is cost and predictability. Hierarchical uses more LLM calls because the manager reasons at every step. And the output path is harder to debug because you did not design it. For production systems, start with sequential, measure quality, and only switch to hierarchical if you see the quality ceiling.

---

PART C: Memory, Delegation &amp; Guardrails

Three more features you should know about before the lab.

Memory. CrewAI can give agents short-term memory (within a task), long-term memory (across crew runs), and entity memory (facts about specific things). Turn it on with memory=True on the Crew. It makes a real difference when agents need to build on each other's work.

Delegation. In hierarchical mode, agents can delegate sub-tasks to other agents. The manager says "Agent B, handle this piece" and Agent B runs its own mini-workflow. You control this with allow_delegation=True on the agent.

Guardrails. You can set max_iter to limit how many reasoning steps an agent takes, and max_rpm to throttle API calls. Without these, a confused agent can loop forever and burn through your API budget. Always set limits.

Tools, processes, memory, delegation, guardrails. That is the full CrewAI toolkit. You have everything you need for the next lab.

Video 2: Gradio -- Instant AI Demos

PART A: Gradio Origins

Gradio was created by Abubakar Abid in 2019 and acquired by Hugging Face in 2021. The idea is simple: take any Python function and turn it into a web interface automatically.

Here is how it works. You write a function. You wrap it in gr.Interface. You call launch. That is it. Gradio looks at your function signature, creates input components for the parameters, creates an output component for the return value, and gives you a local URL.

---

PART B: The Share Link Magic

But here is what makes this powerful: add share=True and Gradio creates a public URL. Anyone in the world can use your app. No deployment. No server. Just a link.

Gradio has components for everything: text, images, audio, video, files, code, chatbots. You can build a text analyzer, a chatbot, an image classifier. All with the same pattern.

Video 3: Streamlit -- AI Dashboards

PART A: Streamlit Philosophy

Streamlit was created in 2018 by Adrien Treuille, Amanda Kelly, and Thiago Teixeira. It takes a different approach than Gradio. Instead of wrapping functions, you write a script. The script runs top to bottom. Every Streamlit call adds something to the page.

st.title adds a title. st.write adds text. st.line_chart adds a chart. st.text_input adds an input box. It reads like English.

---

PART B: State and Caching

What makes Streamlit powerful for AI is its session state and caching. Session state remembers user inputs across reruns. Caching prevents expensive API calls from running every time the page refreshes. Together, they make it possible to build responsive AI dashboards.

---

PART C: Deployment

Streamlit also has a sidebar, columns, tabs, and metrics. Those are all the building blocks of a professional dashboard. And Streamlit Community Cloud lets you deploy for free. Push to GitHub, connect Streamlit Cloud, and your app is live.

If Gradio is for demos, Streamlit is for products. Both are essential tools in your toolkit.

MODULE M10: MCP & GUARDRAILS
================================================================

Video 1: "The Spark" -- The USB of AI

PART 01: The USB Analogy

Think about USB. Before USB, every device had its own proprietary connector. Printers, cameras, keyboards. All different cables. USB created one standard.

MCP, the Model Context Protocol, is doing the same thing for AI. One standard way for any AI model to connect to any tool, any database, any service.

---

PART 02: Before MCP

Before MCP, every integration was custom. Want your AI to read from a database? Write a custom tool. Want it to search the web? Write another tool. Want it to send emails? Another tool. Every integration, from scratch.

MCP changes that. It defines a standard protocol. A server exposes capabilities, a client connects to them. Any AI model can connect to any MCP server. OpenAI adopted it. Google adopted it. Anthropic created it. The entire industry is converging on this one standard.

---

PART 03: Connection + Safety

But connection without safety is a liability, not a feature. If your AI can read from a database, it can also leak data. If it can send emails, it can send harmful ones. If it can execute code, it can execute malicious code.

That is why this module covers both MCP and guardrails. Connection and safety. Power and responsibility. You need both.


Video 2: Building MCP Servers

PART A: MCP Architecture

An MCP server is a program that exposes tools to AI models through a standard protocol. Here is how to build one.

The architecture is simple. You have a server that registers tools. These are functions with names, descriptions, and parameter schemas. You have a client, your AI application, that discovers those tools and calls them.

---

PART B: Building with the MCP SDK

In Python, you use the MCP SDK. You create a server, decorate functions as tools, and run it. The SDK handles the protocol. The JSON-RPC messages, the capability negotiation, the error handling.

Here is what makes this powerful. Once you build an MCP server, any MCP client can use it. Claude Desktop, Cursor, your own applications. They all speak the same protocol. Build once, connect everywhere.

---

PART C: Lab Preview -- Database MCP

In the lab, you will build an MCP server that connects AI to a SQLite database. The AI will be able to query tables, run SQL, and get results. All through MCP. No custom tool code. Just the standard protocol.


Video 3: Implementing Guardrails

PART A: Three Layers of Safety

Guardrails are the safety systems that keep AI applications from going off the rails. There are three layers you need.

Layer one: input validation. Before any user input reaches the LLM, you validate it. Check for prompt injection attacks, someone trying to override the system prompt. Check for prohibited content. Check for excessive length. This is your first line of defense.

---

PART B: Output Filtering

Layer two: output filtering. After the LLM generates a response, you check it. Does it contain harmful content? Does it reveal system prompts or internal data? Does it include hallucinated facts that could be dangerous? You filter before the user sees it.

---

PART C: Behavioral Constraints

Layer three: behavioral constraints. These are the rules your AI follows. It will not execute arbitrary code. It will not access data outside its scope. It will not make claims about topics it should not discuss. These constraints are embedded in the system prompt and enforced in code.

---

PART D: Frameworks

The frameworks make this easier. Guardrails AI, NeMo Guardrails from NVIDIA, and OpenAI's moderation API all provide pre-built safety checks. In the lab, you will implement all three layers.

For years, AI was a text-only game. You type, it responds. But something shifted. The latest models from OpenAI, Google, and Anthropic are now multimodal. They process images, audio, even video. They do not just read your question about a chart. They look at the chart.

Video 2: Vision APIs -- Sending Images to AI

PART A: How to Send Images

Sending an image to an AI model is simpler than you think.

With OpenAI's multimodal models, you send a message that contains both text and an image. The image can be a URL or a base64-encoded file. The model processes both together and responds.

With Gemini, the approach is similar but the API structure differs slightly. You can send images, PDFs, even short video clips.

---

PART B: What Vision Models Can Do

Vision models can do more than you might expect. Object detection, text extraction from photos, diagram analysis, comparing two images, describing scenes, reading handwriting. The accuracy varies by task, but for many practical applications it is remarkably good.

---

PART C: Conversational Vision

The key insight is this. Vision is not a separate capability. It is integrated into the same conversation. You can ask follow-up questions about an image. You can send multiple images and ask the model to compare them. It is a conversation, not a one-shot analysis.


Video 3: LLM Evaluation -- Metrics and Frameworks

PART A: The Hard Problem

You have built a RAG system. You have built agents. But here is the question nobody asks until it is too late. How do you know if they are working well?

LLM evaluation is one of the hardest problems in AI engineering. Traditional software has unit tests. Input X, expect output Y. But LLMs are non-deterministic. The same input can produce different outputs every time.

---

PART B: Automated Metrics

Evaluation starts with knowing what to measure. There are automated metrics like BLEU and ROUGE. They compare generated text against reference text. Useful for translation and summarization. There is semantic similarity, embedding the output and comparing it to expected answers. And there is human evaluation, having people rate quality, accuracy, helpfulness.

---

PART C: RAG Evaluation with RAGAS

For RAG systems specifically, you measure retrieval quality (did we find the right documents?) and generation quality (did the answer correctly use those documents?). Frameworks like RAGAS give you metrics for both.

---

PART D: Practical Approach -- Regression Testing for AI

The practical approach? Build a test suite. A set of questions with known good answers. Run your system against it regularly. Track the scores over time. That is regression testing for AI.

PART A: Built-In Tools

The Claude Agent SDK is Anthropic's official toolkit for building AI agents. Here is what makes it different.

PART B: Skills and Subagents

Second, Skills versus Tools. Tools are atomic actions. Skills are higher-level capabilities. Reusable instruction sets that the agent can invoke. Think of tools as functions and skills as libraries.

The core concept is the Agent class. You define an agent with a name, model, instructions, and tools. Tools are Python functions decorated with metadata. The agent calls them as needed.

PART B: Multi-Agent Orchestration

What makes ADK interesting is multi-agent orchestration. You can define multiple agents that collaborate on complex tasks. A research agent gathers information. An analysis agent processes it. A writing agent produces the final output.
