Video 2: "What Is Claude Code"

Voiceover + animation -- ~5 minutes -- 3 parts
PART A Claude Code Overview
Voiceover + Animation ~2 min

Claude Code is Anthropic's AI-powered coding tool. It comes in two forms: a command-line interface you run in your terminal, and an extension for VS Code and JetBrains IDEs. Both connect to Claude, Anthropic's large language model, but they do much more than send prompts to an API.

Claude Code reads your codebase. It navigates your files, understands your directory structure, and builds a mental model of how your project fits together. It writes code, creates new files, edits existing ones. It runs terminal commands, executes tests, installs packages. It interacts with git, creating commits and managing branches.

This is not autocomplete. Autocomplete predicts the next few characters. Claude Code reasons about your entire project and takes action. The difference is like comparing a spell checker to a co-author.

Storyboard -- Claude Code Architecture
MEDIA TEAM: Animate capabilities appearing one by one around the central Claude Code box. Each lights up as the voiceover names it. End with the autocomplete vs Claude Code comparison fading in at the bottom.
PART B How It Works
Voiceover + Animation ~1.5 min

When you start a Claude Code session, the first thing it does is build context. It looks at your project structure, your open files, your recent git history, and any errors in your terminal. It does not just see the file you are working on. It sees the whole project.

That context is the key insight. A traditional coding assistant sees one file at a time. Claude Code sees how your files connect. It knows that changing a function signature in one file means updating every caller in three other files. It knows that your test suite expects a specific return type. It reasons about the system, not just the syntax.

Then it acts. It proposes changes, explains why, and waits for your approval. If you accept, it makes the edits, runs the tests, and iterates if something breaks. The loop is tight: propose, approve, execute, verify.

Storyboard -- Context Building
MEDIA TEAM: Side-by-side comparison. Left side is muted and dim. Right side is bright with connection lines animating between files. The contrast should feel like going from a flashlight to floodlights.
PART C The Permission Model
Voiceover + Animation ~1.5 min

Claude Code can do a lot. It can edit your files, run commands, push to git. That is powerful, and power needs guardrails.

Every time Claude Code wants to take an action, it asks permission. It shows you exactly what it plans to do: the file it wants to edit, the command it wants to run, the commit message it wants to use. You approve or reject each action individually. Nothing happens without your explicit consent.

You can also configure trust levels. Allow all file reads but require approval for file writes. Allow test execution but block pushes to remote repositories. The default is conservative: ask before everything. You relax the controls as you build confidence.

This is the right design. An AI tool that modifies your code without asking is not a partner. It is a liability. Claude Code treats your codebase with the same respect a good colleague would. It proposes. You decide.

Video 3: "Working with Claude Code"

Voiceover + screen recording -- ~5 minutes -- 3 parts
PART A Starting a Session
Voiceover + Screen Recording ~2 min

You open your terminal, navigate to your project directory, and type claude. That is it. Claude Code starts, scans your project, and waits for instructions.

The quality of your first message determines the quality of the entire session. A vague request like "fix my code" gives Claude Code nothing to work with. A specific request like "the login endpoint returns a 500 error when the email field is empty, find the bug and add input validation" gives it everything it needs.

This is prompting, applied to software engineering. The same skills you learned earlier in this course apply here. Be specific. Provide context. State what you want, not how to get there. Claude Code will figure out the how.

One more thing. If your project has a README or documentation, Claude Code reads it. If it has test files, Claude Code understands your testing patterns. The more structure your project has, the better Claude Code performs. Good engineering practices make AI tools more effective. That is not a coincidence.

PART B Common Workflows
Voiceover + Screen Recording ~1.5 min

Four workflows cover ninety percent of what you will use Claude Code for.

Debugging. Paste the error, point to the file, and say "this test is failing, help me understand why." Claude Code reads the test, reads the source, traces the logic, and explains the bug. Often it fixes it in the same step.

Building features. Describe the feature in plain English. "Add a login page with email and password fields, validate inputs, hash the password, store the user in the database." Claude Code creates the files, writes the routes, adds the templates, and wires everything together.

Refactoring. "This function is 200 lines long. Break it into smaller functions with clear names." Claude Code reads the function, identifies logical boundaries, extracts helper functions, updates all callers, and runs the tests to make sure nothing broke.

Code review. "Review this pull request for bugs, security issues, and style problems." Claude Code reads every changed file, compares against the base branch, and gives you a structured review with line-specific comments.

Storyboard -- Four Core Workflows
MEDIA TEAM: Four cards, each highlighting as the voiceover covers that workflow. Alternatively, show a brief screen recording snippet for each (2-3 seconds per workflow).
PART C Advanced Patterns
Voiceover + Screen Recording ~1.5 min

Three features take Claude Code from useful to indispensable.

CLAUDE.md files. You create a file called CLAUDE.md in your project root. It contains instructions for Claude Code: your coding conventions, your preferred libraries, things it should never do. Claude Code reads this file at the start of every session. Think of it as onboarding documentation for your AI partner.

Memory. Claude Code remembers context across sessions. It stores project-specific notes, your preferences, and patterns it has learned about your codebase. You do not have to re-explain your project every time you start a new session. The memory builds over time, and the tool gets better the more you use it.

Hooks and custom commands. You can define hooks that run automatically before or after certain actions. Format code before every commit. Run linting after every file edit. You can also create custom slash commands that bundle complex workflows into a single trigger. This is where Claude Code becomes a true extension of your engineering practice, not just a tool you use occasionally.

Open a terminal in a sample project directory and start a Claude Code session. Record a live screen walkthrough demonstrating key workflows. Talk naturally — no script. Friendly, patient, at a pace a first-time student can follow.

2. Demonstrate Debugging

3. Build a Small Feature

4. Show CLAUDE.md

  • Point to the exercise for this module
  • Tell students what to submit and where

Video 5: "Building a Real Project"

Prof on camera -- ~3 minutes -- 2 parts
PART A The Workflow, Step by Step
On Camera ~2 min

Say you want to build a FastAPI backend with user authentication. From scratch. No boilerplate, no starter template. Just an empty directory and Claude Code.

Step one: you describe the project. "Build a FastAPI app with user registration, login, JWT authentication, and a protected endpoint that returns the current user's profile." That is your entire specification.

Claude Code starts by creating the project structure. A main application file, a models file for the database schema, a routes file for the endpoints, a config file for settings. It writes the code for each file, installs the dependencies, sets up the database migration.

Then it runs the tests. If something fails, it reads the error, fixes the code, and runs the tests again. You watch, approve the changes, and occasionally steer the direction. "Use bcrypt for password hashing." "Add rate limiting to the login endpoint." Each instruction refines the output.

Within fifteen minutes, you have a working application with authentication, tests, and clean code. Not perfect code. Not production-ready code. But a solid foundation that would have taken you hours to build from scratch.

MEDIA TEAM: Animated timeline that builds from top to bottom as Prof narrates each step. Each node lights up and a brief code snippet or file tree appears beside it. The final node should glow green to signal completion.
PART B Know the Boundary
On Camera ~1 min

Claude Code excels at boilerplate. CRUD endpoints, authentication flows, database schemas, configuration files. It writes these faster and more consistently than most developers. It excels at debugging because it can read error messages, trace the stack, and check multiple files simultaneously. And it excels at exploring unfamiliar codebases because it can scan thousands of files in seconds and build a map of how everything connects.

It struggles with novel algorithms. If you are inventing a new optimization technique or designing a custom data structure, Claude Code will give you something that looks plausible but may not be correct. It struggles with highly creative design decisions. Choosing the right architecture for a system, deciding which trade-offs to make, knowing when simplicity beats sophistication. Those are human judgments.

The engineers who get the most from Claude Code are the ones who use it for what it is good at and keep the hard thinking for themselves.

Claude Code does not replace the engineer. It removes the friction. You still design the system. You still make the decisions. You still own the architecture, the trade-offs, and the consequences. But the distance between an idea and working code has never been shorter.
