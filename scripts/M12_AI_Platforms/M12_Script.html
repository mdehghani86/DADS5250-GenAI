Video 2: "Claude Platform"

Voiceover + animation -- ~5 minutes -- 3 segments
PART A Claude API Basics
Voiceover + Animation ~2 min

Anthropic takes a different stance than the rest of the industry. Their founding principle is building AI that is safe and steerable. That philosophy shows up in everything, from the way Claude handles sensitive prompts to the way the API is designed.

The Claude API uses a messages format. You send a list of messages with roles: user and assistant. System prompts go in a separate system parameter, not inside the messages array. This is a small difference from OpenAI, but it matters when you are debugging.

import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system="You are a helpful data analyst.",
    messages=[
        {"role": "user",
         "content": "Summarize Q3 revenue trends."}
    ]
)

print(response.content[0].text)

Notice the response structure. Claude returns a content array, not a single string. Each element has a type. For text responses, the type is "text" and the value is in the text field. This matters when tool use enters the picture, because tool calls show up as a different content type in that same array.

Storyboard -- Claude API Flow
PART B Claude Tool Use
Voiceover + Code ~2 min

Tool use is where Claude becomes more than a text generator. You define tools with a name, a description, and a JSON schema for the input parameters. When Claude decides it needs to call a tool, it does not return text. It returns a content block with type "tool_use" that includes the tool name and the arguments.

Your code executes the tool, then sends the result back as a message with role "user" and a content block of type "tool_result." Claude reads the result and formulates its final answer. This is a two-turn loop: Claude asks to use a tool, you run it, you send back the result.

# Define a tool
tools = [{
    "name": "get_stock_price",
    "description": "Get current stock price",
    "input_schema": {
        "type": "object",
        "properties": {
            "ticker": {"type": "string"}
        },
        "required": ["ticker"]
    }
}]

# Claude returns: type="tool_use"
# You execute, then send back: type="tool_result"

The key insight: Claude never executes anything. It proposes a tool call. Your code is always in the loop. That is the safety model. The LLM decides what to call. You decide whether to actually run it.

Storyboard -- Tool Use Loop
PART C Claude Agent SDK
Voiceover + Animation ~1 min

Anthropic also ships a Claude Agent SDK for building agentic applications in Python. If you used Claude Code earlier in this course, you saw what a fully built agent looks like. The SDK is the foundation underneath it.

The SDK gives you an agent loop, tool registration, and conversation management out of the box. You define your tools, configure the agent, and the SDK handles the back-and-forth of tool calls and responses automatically. It is the difference between building an agent from raw API calls versus using a framework that manages the loop for you.

For most production use cases, you start with the raw API to understand how it works, then graduate to the SDK when you need persistent agents with multiple tools.

Video 3: "OpenAI Platform"

Voiceover + animation -- ~5 minutes -- 3 segments
PART A OpenAI API and Function Calling
Voiceover + Code ~2 min

OpenAI is the platform most of you started with. The chat completions endpoint is the foundation. You send messages with roles: system, user, and assistant. You get back a response with choices, each containing a message.

Function calling is OpenAI's version of tool use. You define functions with a name, description, and JSON schema for the parameters. When the model decides to call a function, the response includes a tool_calls array instead of regular content. You execute the function, then send the result back as a message with role "tool."

from openai import OpenAI

client = OpenAI()

response = client.chat.completions.create(
    model="gpt-4.1-mini",
    messages=[
        {"role": "system", "content": "You are a data analyst."},
        {"role": "user", "content": "Summarize Q3 revenue."}
    ],
    tools=[{
        "type": "function",
        "function": {
            "name": "get_revenue",
            "description": "Fetch quarterly revenue",
            "parameters": {
                "type": "object",
                "properties": {
                    "quarter": {"type": "string"}
                }
            }
        }
    }]
)

OpenAI also supports structured outputs. You can pass a response_format with a JSON schema, and the model guarantees its output matches that schema exactly. No parsing errors. No malformed JSON. This is extremely useful for production pipelines where downstream code depends on a specific data shape.

PART B OpenAI Agents SDK
Voiceover + Code ~2 min

In early 2025, OpenAI released the Agents SDK. This is their answer to the question: how do you build multi-step, multi-agent workflows without writing all the orchestration code yourself?

The SDK introduces three core concepts. An Agent is a configured LLM with instructions, tools, and optional guardrails. A Handoff lets one agent transfer control to another agent when the task requires different expertise. And a Runner executes the whole workflow, managing the agent loop, tool calls, and handoffs automatically.

from agents import Agent, Runner

research_agent = Agent(
    name="Researcher",
    instructions="Find relevant data sources.",
    tools=[search_tool]
)

writer_agent = Agent(
    name="Writer",
    instructions="Write a clear summary.",
    handoffs=[research_agent]
)

result = Runner.run_sync(
    writer_agent,
    "Summarize AI trends for Q3."
)

The handoff mechanism is worth understanding. When the writer agent realizes it needs data, it hands off to the research agent. The research agent does its work and returns. The writer picks up where it left off. No manual orchestration. No custom state management. The SDK handles the transitions.

Guardrails are built into the SDK as well. You can define input and output validators that run automatically before and after each agent turn. If a guardrail fails, the agent stops. This is production-grade safety without extra infrastructure.

Storyboard -- OpenAI Agents SDK
PART C Assistants API
Voiceover + Animation ~1 min

OpenAI also offers the Assistants API, which takes a different approach. Instead of stateless chat completions, the Assistants API manages persistent threads. You create an assistant with instructions and tools. You create a thread for a conversation. You add messages to the thread and create a run. The assistant processes the whole thread and responds.

The Assistants API includes two powerful built-in tools. File search lets the assistant search through uploaded documents. Code interpreter lets it write and execute Python code in a sandboxed environment. These tools run on OpenAI's infrastructure. You do not host anything.

For applications that need persistent conversations with file handling and code execution, the Assistants API is a strong choice. For lightweight, stateless workflows, the chat completions endpoint or the Agents SDK is simpler.

Video 4: "Google Platform"

Voiceover + animation -- ~5 minutes -- 3 segments
PART A Gemini API
Voiceover + Code ~2 min

Google's Gemini is multimodal from the ground up. Text, images, video, audio, and code all go through the same model. You do not need separate endpoints for vision versus text. You send a mixed content array, and Gemini handles it.

The practical advantage is the free tier. Google AI Studio gives you access to Gemini models with generous rate limits at no cost. For students and small projects, this is significant. You can prototype entire applications without a credit card.

import google.generativeai as genai

genai.configure(api_key="YOUR_API_KEY")

model = genai.GenerativeModel("gemini-2.5-flash")

# Text + image in one call
response = model.generate_content([
    "Describe what you see in this chart.",
    image_data  # PIL Image or bytes
])

print(response.text)

The API design is simpler than OpenAI or Claude for basic use cases. You create a model object, call generate_content, and get back a response with a text property. For multimodal inputs, you pass a list with text and images mixed together. No special configuration. No separate vision model.

Google AI Studio also provides a web playground where you can test prompts, upload files, and experiment with system instructions before writing any code. Think of it as Google's version of the OpenAI Playground.

Storyboard -- Gemini Multimodal
PART B Google ADK (Agent Development Kit)
Voiceover + Code ~2 min

Google released the Agent Development Kit, or ADK, in 2025. It is Google's framework for building AI agents that can use tools, maintain state, and collaborate with other agents.

The core abstraction is an Agent class. You give it a name, a model, instructions, and a list of tools. The ADK handles the agent loop: the model generates a response, the ADK checks if it includes tool calls, executes them, and feeds the results back. Same pattern you saw with Claude and OpenAI, just different syntax.

from google.adk import Agent

agent = Agent(
    name="data_analyst",
    model="gemini-2.5-flash",
    instruction="Analyze data and provide insights.",
    tools=[query_database, create_chart]
)

# Run the agent
response = agent.run(
    "What were the top products last quarter?"
)

The ADK supports sub-agents, which are agents that can delegate to other agents. This is Google's approach to multi-agent orchestration. You define a parent agent and give it child agents as tools. When the parent needs specialized work, it delegates to the right child, gets the result, and continues.

One advantage of the ADK is tight integration with Google Cloud services. If your stack already runs on Google Cloud, the ADK gives you seamless access to BigQuery, Vertex AI, and other services without extra configuration.

Storyboard -- Google ADK Architecture
PART C A2A Protocol
Voiceover + Animation ~1 min

Google also introduced the A2A protocol. A2A stands for Agent-to-Agent. It is an open standard for agents built on different platforms to communicate with each other.

Think about it this way. Right now, a Claude agent and a Gemini agent cannot easily talk to each other. They live in separate ecosystems with different APIs. A2A defines a common protocol so agents can discover each other, exchange messages, and coordinate work regardless of which platform built them.

This is still emerging. The specification was published in 2025 and adoption is in early stages. But the direction is important. The future of AI agents is not one platform wins. It is agents from different platforms working together. A2A is Google's bet on that future.

Video 5: "Comparison and When to Choose"

Prof on camera -- ~3 minutes -- 2 segments
PART A Side-by-Side Comparison
On Camera + Overlay ~2 min

Now that you have seen all three, the patterns should be obvious. Every platform has messages. Every platform has tool calling. Every platform has an agent framework. The differences are in the details.

Claude separates the system prompt from messages. OpenAI puts it inside messages. Gemini uses a model-level system_instruction. Claude returns tool calls as content blocks in a content array. OpenAI returns them in a tool_calls array inside the message. Gemini uses function_call parts. Different shapes, same idea.

On the agent side: Claude has the Agent SDK, OpenAI has the Agents SDK with handoffs and guardrails, Google has the ADK with sub-agents and Google Cloud integration. Each framework reflects the priorities of the company that built it.

For pricing, Claude and OpenAI charge per token with different tiers. Gemini offers a generous free tier through Google AI Studio. For production, all three offer enterprise pricing with SLAs. The cost difference for most applications is small enough that it should not be the primary decision factor. Capability and ecosystem fit matter more.

Storyboard -- Comparison Table
PART B The Real Skill
On Camera ~1 min

The real skill is not mastering one platform. It is knowing which one to reach for.

Need bulletproof safety and the best reasoning? Claude. Need the broadest ecosystem with the most third-party integrations? OpenAI. Need multimodal on a budget with tight Google Cloud integration? Gemini.

And the best part: switching is not hard. Once you understand the pattern of messages, tools, and agent loops, moving from one platform to another is a matter of changing import statements and adjusting a few field names. The hard part is the thinking. The syntax is just syntax.
