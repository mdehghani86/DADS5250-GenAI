Video 2: "Google A2A Protocol"

Voiceover + animation -- ~5 minutes -- 3 parts
KEY MESSAGE A2A is the HTTP of agents. A standard protocol so agents built by different teams, in different frameworks, can find each other and collaborate.
PART A What is A2A
Voiceover + Animation ~2 min

A2A stands for Agent-to-Agent. It is an open protocol published by Google in April 2025. The problem it solves is straightforward: agents built by different teams, using different frameworks, running at different companies, have no standard way to talk to each other.

Think about what HTTP did for the web. Before HTTP, every network had its own protocol. After HTTP, any browser could talk to any server. A2A aims to do the same thing for AI agents. Any agent that implements the A2A spec can discover, communicate with, and delegate tasks to any other A2A-compatible agent.

Google did not build this alone. Over 50 technology partners were involved at launch, including Salesforce, SAP, LangChain, and CrewAI. The spec is open. Anyone can implement it.

Storyboard -- The A2A Vision
MEDIA TEAM: Animate the "before" state first (agents isolated, red X between them), then transition to "after" (connections light up green, A2A badge appears). Hold 3 seconds on the final state.
PART B How A2A Works
Voiceover + Animation ~2 min

A2A has three core concepts. Agent Cards, Tasks, and streaming updates.

An Agent Card is like a business card for an AI agent. It is a JSON document hosted at a well-known URL that describes what the agent can do, what inputs it expects, and what authentication it requires. When another agent wants to find a collaborator, it reads the Agent Card first.

Once an agent finds a partner, it sends a Task. A Task is a structured request with a clear goal. The receiving agent processes it and returns results, either all at once or as a stream of updates. Streaming is important because many agent tasks take time, and the calling agent needs to know what is happening.

The flow is: discover the Agent Card, create a Task, receive updates, get the final result. Four steps. That is the entire protocol.

Storyboard -- A2A Flow
MEDIA TEAM: Animate the four steps left-to-right, each box lighting up in sequence. Then fade in the Agent Card JSON below. Highlight each field as the narrator names it.
PART C Why A2A Matters
Voiceover + Animation ~1.5 min

The real value of A2A is interoperability. Right now, if you build an agent in CrewAI, it only talks to other CrewAI agents. If someone else builds an agent in LangGraph, those two systems cannot collaborate without custom glue code.

A2A changes that. Your CrewAI research agent publishes an Agent Card. Someone else's LangGraph analysis agent reads that card, sends a task, and gets results back. The frameworks do not matter. The protocol handles the translation.

This is how enterprise AI will work. Not one company building every agent, but many teams building specialized agents that discover and collaborate with each other through a shared standard. A2A is that standard.

Video 4: "OpenAI Agents SDK"

Voiceover + animation -- ~5 minutes -- 3 parts
KEY MESSAGE The OpenAI Agents SDK gives you Agent, Handoff, and Runner. Three building blocks, and you can wire up a multi-agent system in under 50 lines of Python.
PART A OpenAI Agents SDK Overview
Voiceover + Animation ~2 min

OpenAI released the Agents SDK in March 2025. It is a lightweight Python framework for building agentic applications. The philosophy is minimalism: give developers a small number of powerful primitives and let them compose.

There are three core concepts. Agent: a configured LLM with instructions, tools, and optional guardrails. Handoff: a mechanism for one agent to transfer control to another agent. Runner: the execution engine that takes an agent and a task, runs the loop, and returns the result.

That is the entire surface area. Agent, Handoff, Runner. If you understand those three, you understand the SDK.

The SDK also includes built-in guardrails. You can define input validators and output validators that run before and after every agent turn. If a guardrail fails, the agent stops. No hallucinated output reaches the user. This is production thinking baked into the framework from day one.

Storyboard -- SDK Architecture
MEDIA TEAM: Animate the three boxes appearing one at a time (Agent, Handoff, Runner). Then the guardrails bar slides in from below. Finally, the code snippet fades in at the bottom.
PART B The Code Pattern
Voiceover + Code ~2 min

Building a multi-agent system with the SDK follows a clean pattern. You define your agents, each with a name, instructions, and tools. You wire handoffs so agents know which other agents they can transfer to. Then you run the whole thing with Runner.

A triage agent receives the user request and decides which specialist handles it. A research agent searches the web. A writing agent drafts the response. The triage agent hands off to the right specialist based on the request, and Runner manages the execution loop until the task is complete.

The handoff is the key innovation. Unlike CrewAI where you define the entire crew up front, here each agent declares who it can hand off to. The routing is dynamic. The triage agent decides at runtime which specialist gets the job.

Storyboard -- Multi-Agent Handoff
MEDIA TEAM: Animate the triage agent receiving a request, then a dashed arrow shooting down to one of the specialists. Show the Runner bar pulsing at the bottom to indicate it is managing the loop.
PART C Assistants API (Brief)
Voiceover + Animation ~1.5 min

OpenAI also offers the Assistants API, which takes a different approach. Instead of running everything locally with the SDK, the Assistants API is a managed, server-side service. You create an Assistant, start a Thread, add Messages, and trigger a Run. OpenAI's servers handle the execution.

The Assistants API includes built-in tools like File Search and Code Interpreter. File Search indexes your documents and lets the assistant answer questions about them. Code Interpreter gives the assistant a sandboxed Python environment to write and execute code on the fly.

The trade-off is control versus convenience. The Agents SDK runs locally, you own the execution loop, you can customize everything. The Assistants API is managed, easier to set up, but you are relying on OpenAI's infrastructure. For this course, we focus on the SDK because it teaches you the mechanics. But knowing the Assistants API exists gives you a production option when speed matters more than customization.
