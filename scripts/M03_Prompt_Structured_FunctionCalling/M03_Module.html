Every morning I place the same coffee order. Medium dark roast, whole milk on the side, cinnamon on top. My barista knows it by heart. I never have to explain myself twice.

That is exactly the standard we should hold ourselves to when we communicate with AI models.

In Module 2, you built an Excel GPT and a Bitcoin analyzer. They worked -- mostly. But sometimes the output was inconsistent. Sometimes the model gave you a paragraph when you needed a formula.

This module fixes that. You will learn three levels of control. First -- how to write prompts that get exactly the response you need. Second -- how to force the output into structured JSON that code can use. Third -- how to make the model decide when to call a function and take action.

By the end of this module, your AI outputs will be reliable, structured, and actionable. Every single time.

Let us start with the most fundamental concept: how you talk to an AI model through code.

Every API call has three layers. The system message sets the persona -- who the model is. The user message is your question or request. And the assistant message is what the model generates back.

Let me show you something. Same question, two different system messages. With a "You are a helpful assistant" system message, the model gives you a general answer. With "You are a senior Python developer who writes production-ready code" -- the exact same question produces code with error handling, type hints, and documentation.

That is the power of the system message. Same engine, different driver.

Now, every good prompt has four essential elements. Role -- who is the model? Task -- what should it do? Context -- what does it need to know? Format -- how should the output look?

Let me show you a bad prompt versus a good prompt. Bad: "Tell me about supply chains." Good: "You are a supply chain consultant. Analyze the bullwhip effect for a mid-size electronics manufacturer. Provide 3 mitigation strategies, each with expected cost reduction percentage. Format as a numbered list."

Same model. Dramatically different output quality. The difference is entirely in the prompt.

Now that you know the anatomy of a prompt, let us talk about strategies -- the techniques that make prompts consistently reliable.

Strategy one: zero-shot. You give the model a task with no examples. "Classify this review as positive, negative, or neutral." It works most of the time. But what about edge cases? "The food was amazing but the service ruined everything." Is that positive or negative? The model might say either one.

Strategy two: few-shot. You give the model a few examples before the actual task. "Here are three reviews with their classifications. Now classify this one." Suddenly the edge case gets consistent results. The model learns the pattern from your examples.

Strategy three: chain-of-thought. You tell the model to think step by step. Here is the classic example. "A bat and a ball cost $1.10 together. The bat costs $1.00 more than the ball. How much does the ball cost?"

Without chain-of-thought, most models say 10 cents. That is wrong. With chain-of-thought -- "Think step by step" -- the model works through the algebra and gets 5 cents. Just four words improved accuracy by 40% on reasoning tasks.

When do you use which? Zero-shot for simple tasks. Few-shot when you need consistent formatting. Chain-of-thought when the task requires reasoning. You will practice all three in Lab 1.

You now control what the model says and how it behaves. But there is still a problem.

The output is text. Free-form, unstructured text. And text is great for humans -- but terrible for code.

You cannot put a paragraph into a database. You cannot feed prose into an analytics pipeline. You need structure. You need JSON.

JSON mode forces the model to return valid JSON -- guaranteed parseable, guaranteed structured.

Let me show you the difference. Same question, two modes.

Text mode: "The sentiment of the review is positive. The customer expressed satisfaction with the product quality and delivery speed."

JSON mode: {"sentiment": "positive", "confidence": 0.92, "keywords": ["quality", "delivery speed"]}

Same information. But the JSON version can be parsed by code in one line. No regex. No string manipulation. Just json.loads() and you have a Python dictionary.

But JSON mode has a gap. It guarantees valid JSON, but it does not guarantee the right fields or the right types. That is where Pydantic comes in.

Pydantic lets you define a schema -- a class that says "I expect a field called sentiment of type string, a field called confidence of type float between 0 and 1." If the model returns something that does not match, Pydantic catches it immediately. No silent bugs. No bad data reaching your database.

Until now, the model only produces text -- or structured text. Function calling lets it produce actions.

You define tools -- functions with names, descriptions, and parameter schemas. The model reads the user request and decides: which tool should I use? What arguments should I pass?

The model does not execute the function. It generates a structured call -- a JSON object saying "call this function with these arguments." Your code handles execution.

This is the bridge between a chatbot and an agent. The model reasons about which tool to use. Your code does the work.

Let me walk through a complete example.

Step 1: You define a function called get_weather with parameters city and units.

Step 2: User says "What is the weather in Boston?"

Step 3: The model does not answer directly. Instead it returns: {"name": "get_weather", "arguments": {"city": "Boston", "units": "fahrenheit"}}

Step 4: Your code calls the real weather API with those arguments.

Step 5: You send the result back to the model. Now it generates a human-readable response: "It is currently 72 degrees in Boston with partly cloudy skies."

Five steps. The model decided which function to call. Your code did the work. That is function calling.

This was the biggest module in the course -- and for good reason.

You learned to engineer prompts that produce reliable output. You learned to force that output into structured JSON and validate it with Pydantic. And you learned to let the model decide when to call functions and take actions.

That is the complete pipeline. Input control. Output control. Action control.

Think about what you can build now. An AI that reads invoices and validates the data before it touches your database. An AI that answers questions by querying a real database. An AI assistant with tools.

Next module -- we chain these operations into multi-step workflows. That is LangChain. And we put it to the test with a supply chain simulation called the Beer Game.

The Anatomy of a Prompt

Every API call follows a three-layer framework. Understanding these layers is the foundation of prompt engineering.

The Three-Layer Framework

Every message you send to an LLM API has three possible roles. Together they form the complete communication structure.

System Message (Blue Layer)
Sets the persona, rules, and constraints
"You are a senior data analyst. Always respond with data-backed insights. Use bullet points."
User Message (Green Layer)
Your question, request, or instruction
"Analyze Q4 2025 revenue trends for our e-commerce platform."
Assistant Message (Gray Layer)
The model's response (or a previous response in multi-turn)
"Based on the data, Q4 showed a 12% increase in revenue driven by..."
IMAGE: Screenshot of ChatGPT showing how a system message changes output for the same question

The Four Essential Elements

Every effective prompt contains these four elements -- whether you include them explicitly or not.

1. Role

Who is the model? "You are a supply chain consultant with 15 years of experience."

2. Task

What should it do? "Analyze the bullwhip effect and provide mitigation strategies."

3. Context

What does it need to know? "The company is a mid-size electronics manufacturer with 3 distribution centers."

4. Format

How should output look? "Return a numbered list with cost reduction percentages."

The Bad Prompts Museum

A gallery of terrible prompts -- and how to fix every one of them.

"If your prompt could be about anything, the output will be about nothing."

Bad Prompt
"Tell me about data"
Result: Generic 500-word essay about data in the modern world
The Fix
"You are a data engineer. Compare 3 data lakehouse architectures (Databricks, Snowflake, BigQuery) for a 50TB retail dataset. Include cost per TB/month."
Result: Structured comparison with actionable cost data
Bad Prompt
"Write code"
Result: Random Python snippet that prints "Hello World"
The Fix
"Write a Python function that takes a CSV file path, reads it with pandas, handles missing values by forward-filling, and returns descriptive statistics as a dictionary. Include type hints and a docstring."
Result: Production-ready function with error handling
Bad Prompt
"Analyze this" (no data attached)
Result: Model makes up fictional data and analyzes it
The Fix
"Here is our Q4 sales data: [paste data]. Identify the top 3 trends and provide a root cause hypothesis for each. If data is insufficient, say so."
Result: Grounded analysis with explicit data references
Bad Prompt
"Be creative"
Result: Unpredictable mess -- sometimes a poem, sometimes a story, sometimes a business plan
The Fix
"Generate 5 product names for a sustainable water bottle brand. Each name should be 1-2 words, evoke nature, and be available as a .com domain. Format: name | tagline."
Result: Focused creative output with clear constraints

Building a Prompt Library

Production teams do not write prompts from scratch. They maintain reusable libraries of tested, versioned templates.

What Is a Prompt Library?

A prompt library is a collection of tested, parameterized prompt templates that your team reuses across projects. Think of it like a code library -- except the "functions" are natural language instructions.

Companies like Stripe, Notion, and Shopify maintain internal prompt libraries. Each template is version-controlled, tested against edge cases, and reviewed before deployment.

4 Example Templates

1. Customer Feedback Analyzer
"Analyze this customer review: {review}. Return JSON with: sentiment (positive/negative/neutral), topics (list), urgency (1-5), suggested_action."
2. Code Reviewer
"Review this {language} code for: bugs, performance issues, security vulnerabilities, and style. Rate severity (low/med/high) for each finding. Code: {code}"
3. Meeting Summarizer
"Summarize this meeting transcript: {transcript}. Output: key_decisions (list), action_items (list with owner + deadline), open_questions (list)."
4. Data Extractor
"Extract structured data from this {document_type}: {text}. Return JSON matching this schema: {schema}. If a field is missing, use null."

Text vs JSON vs Validated JSON

Three levels of output control. Each adds reliability. Production systems need Level 3.

Level 1: Free Text

Unstructured

Human-readable but unparseable. Cannot be used by code without brittle regex or string manipulation.

Reliability: Low
Level 2: JSON Mode

Structured

Guaranteed valid JSON. Parseable by code. But no guarantee of correct fields or types.

Reliability: Medium
Level 3: Pydantic Validated

Validated

Structured + type-checked + error-caught. Wrong types, missing fields, invalid ranges all caught instantly.

Reliability: High

Why Pydantic Matters

JSON mode can return {"price": "twenty dollars"} -- valid JSON, but your code expects a float. Pydantic catches this before it hits your database.

from pydantic import BaseModel class SentimentResult(BaseModel): sentiment: str # must be a string confidence: float # must be a number 0-1 keywords: list[str] # must be a list of strings

Key Takeaways
