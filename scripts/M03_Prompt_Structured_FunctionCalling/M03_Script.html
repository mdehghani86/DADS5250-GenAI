Video 2: "Prompting Foundations"

SEG A The Three-Layer Framework
Voiceover + Animation ~2 min

Every API call you make has three layers. Think of them as three voices in a conversation.

The first is the System layer. This is where you define who the model is -- its role, its rules, its constraints. "You are a data analyst who provides concise, evidence-based responses." This sets the tone for everything that follows.

The second is the User layer. This is your actual request. "Summarize the key findings in this quarterly report." Specific, clear, actionable.

The third is the Assistant layer. This captures the model's previous responses. In multi-turn conversations, this is how the model remembers what it already said.

System, User, Assistant. Every API call you will ever make follows this structure.

Animation Three stacked layers appearing one at a time -- System (blue), User (green), Assistant (gray). Each layer shows its example text. Layers stack into a complete API call. ~20 sec loop

SEG A2 Framework Live Example
Voiceover + Code ~2 min

Let me show you what this looks like in practice.

System message: "You are a customer support agent for an online electronics store. Be friendly and professional. Keep responses under three sentences. If you do not know the answer, say: Let me connect you with a specialist. Never discuss competitor products."

User message: "Hi, I ordered a laptop last week and it still has not arrived. Order number 12345."

Now watch what happens. The model responds within the rules -- friendly tone, under three sentences, asks a follow-up question. That is the system message doing its job.

But here is the key insight -- change the system message to "You are a sarcastic comedian who works at a tech store," and the same user question gets a completely different response. Same input. Different persona. The system message is the control panel.

SEG B Four Essential Elements
Voiceover + Animation ~3 min

Let me show you the difference four elements make.

Bad prompt: "Tell me about renewable energy."

What you get back: a generic, unfocused, 500-word essay that covers everything and nothing.

Good prompt: "You are an energy policy analyst specializing in US manufacturing. Summarize the three most impactful renewable energy technologies for US factories in 2026. For each technology, provide: the name, current adoption rate as a percentage, and one key advantage over fossil fuels. Format your response as a numbered list with bold headings."

What you get back: a focused, structured, actionable summary -- exactly three items, with data, in the format you specified.

The four elements: Role -- energy policy analyst. Task -- summarize three technologies. Context -- US manufacturing in 2026. Format -- numbered list with bold headings.

Same model. Same API call. Dramatically different output. The prompt is the variable.

SEG C1 Zero-Shot Prompting
Voiceover + Code ~2 min

Zero-shot means asking the model to perform a task with no examples -- just instructions.

Here is the prompt: "Classify the sentiment of this restaurant review as positive, negative, or neutral. Reply with just the label. Review: The food was decent but the service was painfully slow."

The model responds: "negative." That worked.

But try an ambiguous review: "The dessert was amazing but I found a hair in my soup."

The model might say "negative" or "mixed." There is no consistency because you gave it no examples of how you define the boundaries. This is where zero-shot breaks down -- on edge cases. And that is exactly when you need few-shot.

Code Display Spec
Code + Response Show the prompt, then the response appearing. Then the ambiguous review -- show the model hesitating between labels. Transition arrow: "Solution -- Few-shot"
MEDIA TEAM: Two-step display. First: clean prompt with "negative" response (green check). Second: ambiguous review with flickering labels -- "negative" / "mixed" / "negative" -- showing inconsistency (amber warning icon). Arrow transition at bottom pointing to next segment: "Solution: Few-shot"
SEG C2 Few-Shot Prompting
Voiceover + Code ~2 min

Few-shot means including examples in your prompt. Watch how this fixes the edge case problem.

Prompt: "Classify the sentiment. Reply with just the label.

Review: Loved the food, great atmosphere! -- positive
Review: Terrible service, cold food, never coming back. -- negative
Review: It was okay. Nothing special. -- neutral
Review: The dessert was amazing but I found a hair in my soup. -- "

Now the model has a pattern. It sees how you handle reviews -- you want a single label, not "mixed." And because the examples show clean single-word labels, the output format is locked in.

The model responds: "negative." Consistent. Formatted correctly. Every time. That is the power of few-shot.

Code Display Spec
Code + Highlighting Show the prompt with examples highlighted in blue. Arrow pointing to the blank last line. Response appears: "negative." Then side-by-side: zero-shot inconsistency vs few-shot consistency.
MEDIA TEAM: Examples in the prompt get a subtle blue highlight. The blank line after the last review pulses gently -- "waiting for response." Then "negative" appears with a green check. Quick side-by-side comparison: zero-shot (flickering labels, amber) vs few-shot (stable label, green). Clean visual proof.
SEG C3 Chain-of-Thought Prompting
Voiceover + Animation ~2 min

Chain-of-thought means telling the model to reason step by step before answering.

Here is the classic test. A bat and a ball cost one dollar and ten cents in total. The bat costs one dollar more than the ball. How much does the ball cost?

Most people -- and most models without CoT -- say ten cents. It feels right intuitively. But it is wrong.

Now add four words: "Think step by step."

The model responds: "Let me work through this. If the ball costs x, the bat costs x plus one dollar. Total: x plus x plus one equals one-ten. So two-x plus one equals one-ten. Two-x equals zero-ten. x equals zero-zero-five. The ball costs five cents."

Four words. Correct answer. Google researchers found that chain-of-thought improved math reasoning accuracy by over forty percent. Four words. Forty percent.

MEDIA TEAM: Left panel (red tint): question appears, instant wrong answer "10 cents" with red X. Right panel (green tint): same question + "Think step by step." then reasoning appears line by line like a typewriter -- each algebraic step. Final answer: "5 cents" with green check. Citation bar at bottom: "Wei et al., Google Research, 2022 -- 40%+ improvement in math reasoning"

The system message is your control panel. It is the one place where you define everything about how the model behaves -- before the user says a single word.

A strong system message includes: the role, behavioral rules, output constraints, and what not to do.

SEG A2 System Message Showcase
Voiceover + Code ~3 min

System message: "You are a senior financial analyst at a Fortune 500 company. Rules: Every claim must include a specific number or percentage. Never say probably or maybe -- either cite data or say insufficient data. Maximum three paragraphs. End every response with one actionable recommendation."

Watch what happens. Ask it: "How is the AI market growing?"

Without this system message, you get a fluffy essay. With it, you get: specific numbers, no hedging, three paragraphs, and an actionable recommendation at the end. The system message turned a chatbot into an analyst.

Now let me show you a second persona. System message: "You are a senior Python developer conducting a code review. Rules: Point out bugs, not style preferences. For each issue, explain why it is a problem and provide corrected code. Rate the overall code quality as A through F. If the code is production-ready, say so."

Paste in a Python function. The model reviews it exactly like a senior engineer would -- prioritizing bugs over style, showing fixes, giving a grade. That structured feedback loop came entirely from the system message.

SEG B Prompt Templates
Voiceover + Code ~2 min

In production, you do not write prompts from scratch every time. You build templates.

A template has fixed structure with variable placeholders. Same skeleton, different inputs.

"You are a {role}. Analyze the following {document_type} for {audience}. Focus on {focus_area}. Return your analysis as {format}."

Now watch me run the same template three times with different variables:

Run 1: role equals data analyst, document type equals quarterly earnings, audience equals executive team, format equals three bullet points.

Run 2: role equals HR manager, document type equals employee survey, audience equals department heads, format equals table.

Run 3: role equals supply chain engineer, document type equals vendor performance data, audience equals procurement team, format equals ranked list.

Same template. Three completely different analyses. That is scalability. That is how production AI systems work.

Animation Spec
Animation Show the template with colored {placeholders}. Three runs -- each fills in the variables with different values, and a different styled output appears. The template stays constant, the outputs change. ~20 sec
MEDIA TEAM: Template at top with placeholders as colored chips: {role} = blue, {document_type} = green, {audience} = amber, {format} = teal. Three runs below: each run fills in the chips with real values and shows a mini output card. The template bar stays fixed at top while runs 1, 2, 3 appear sequentially below. Shows reusability visually.
