1 Introduction to Prompting

When you type a message to ChatGPT, you are writing a prompt. But there is a difference between a prompt that works and a prompt that works well.

A good prompt gets you a useful, accurate, well-structured response on the first try. A bad prompt gets you something vague, generic, or completely wrong -- and then you spend five minutes trying to fix it.

In this video, we are going to learn a simple framework that turns every prompt into a good one. It has four parts. We call it RTCF: Role, Task, Context, Format.

2 The RTCF Framework

Here is the framework. Four letters. Four questions you should ask yourself every time you write a prompt.

R -- Role. Who should the model pretend to be? A data analyst? A teacher? A Python developer? The role sets the tone, the vocabulary, and the expertise level of the response.

T -- Task. What exactly do you want the model to do? Summarize? Compare? Write code? Generate a table? Be specific. "Help me with this" is not a task. "Write a Python function that takes a list of numbers and returns the median" -- that is a task.

C -- Context. What background information does the model need? What data are you working with? What have you already tried? What constraints exist? The more relevant context you give, the better the output.

F -- Format. How should the response look? A bulleted list? A JSON object? A three-paragraph essay? A table with specific columns? If you do not specify the format, the model will guess -- and it will often guess wrong.

3 Role Deep Dive

Let us take each element one at a time. Starting with Role.

When you assign a role, you are not just being cute. You are activating a different distribution of knowledge inside the model. Watch what happens when we ask the same question with different roles.

No role: "Explain what a p-value is." The model gives a generic textbook answer.

Role -- statistics professor: "You are a statistics professor. Explain what a p-value is." Now the response is more precise, uses proper terminology, and might include a formal definition.

Role -- data journalist: "You are a data journalist writing for a general audience. Explain what a p-value is." Now the response is simpler, uses analogies, avoids jargon.

Same question. Different roles. Completely different outputs. That is the power of Role.

4 Task Deep Dive

Now Task. This is where most people fail. They write prompts that are too vague.

Bad task: "Tell me about Python." That could mean anything. The model has to guess what you want, and it will give you a generic overview that helps no one.

Good task: "Write a Python function that reads a CSV file, calculates the average of a specified column, and returns the result as a formatted string." Now the model knows exactly what to build. No guessing. No ambiguity.

The rule is simple: if a human reading your prompt would need to ask a follow-up question, your task is not specific enough.

5 Context Deep Dive

Context is the secret weapon. Most people skip it entirely, and then wonder why the model gives generic answers.

Here is an example. Without context: "Write me a cover letter." The model has nothing to work with. It will produce something bland and forgettable.

With context: "Write me a cover letter. I am a second-year data science graduate student applying for a summer internship at a healthcare analytics startup. I have experience with Python, SQL, and Tableau. The job posting emphasizes collaboration and communication skills." Now the model can write something specific, relevant, and actually useful.

Context is not optional. It is the difference between a generic response and a personalized one.

6 Format Deep Dive

Finally, Format. This is the easiest one to use and the one people forget most often.

Without format instructions, the model defaults to paragraphs of prose. That might be fine for an essay, but terrible for a comparison, a checklist, or structured data.

Examples of format instructions: "Respond in a bulleted list." "Return the result as a JSON object with keys: name, score, recommendation." "Create a markdown table with columns: Metric, Value, Interpretation." "Write exactly three sentences."

Format is not a nice-to-have. When you are building applications that parse LLM output, format is everything. If the model returns a paragraph when your code expects JSON, your pipeline breaks.

7 Bad vs Good Prompt Comparison

Let us put it all together with a real example.

Bad prompt: "Help me with Excel." No role. Vague task. No context. No format. The model will produce something generic and probably useless.

Good prompt: "You are an Excel expert who specializes in financial modeling. Write a VLOOKUP formula that matches product SKUs in column A of Sheet1 to prices in column B of Sheet2. The SKU format is three letters followed by four digits, like ABC1234. Return the formula, a brief explanation of each argument, and one common mistake to avoid. Format as a numbered list."

Role: Excel expert in financial modeling. Task: Write a specific VLOOKUP formula. Context: SKU format, sheet layout, column references. Format: Numbered list with formula, explanation, and common mistake.

That is RTCF. Four elements. One framework. Every prompt you write from now on should have all four.

Video 3: "Prompting Strategies"

Voiceover + animation -- 6-8 minutes -- 4 segments (A intro · B Zero-Shot · C Few-Shot · D Chain-of-Thought)
KEY MESSAGE Zero-shot for simple, few-shot for consistency, chain-of-thought for reasoning.
PART A Intro — Why Strategies Matter
Voiceover + Animation ~45 sec

Now that you know the anatomy of a prompt, let me tell you something that might surprise you. Large language models actually enjoy getting examples. Just like us.

Think about it. If I asked you to "classify these customer reviews" with no context, you would probably ask me to show you a few first. "What does a positive one look like? What does a tricky one look like?" You would want to see the pattern before you start.

LLMs work the same way. Sometimes they can figure it out on their own. Sometimes they need a few examples. And sometimes, for harder problems, they need to slow down and think out loud. These three modes have names: zero-shot, few-shot, and chain-of-thought. In this video, we will walk through all three -- when to use each one, and why.

Storyboard -- Three Strategies Overview
MEDIA TEAM: Three cards fade in one at a time as Prof names each strategy. Zero-Shot (blue), Few-Shot (green), Chain-of-Thought (purple). Final state holds for 2 seconds before transitioning to the next segment.
PART B Zero-Shot Prompting
Voiceover + Animation ~90 sec

Let us start with zero-shot. Zero-shot means exactly what it sounds like -- you give the model a task with zero examples. You just tell it what to do and trust it to figure out the pattern.

Here is a classic zero-shot prompt: "Classify this review as positive, negative, or neutral." No examples. No training. Just the task. And most of the time, this works beautifully -- the model has seen millions of reviews during training, so it knows what positive and negative look like.

But what about edge cases? "The food was amazing but the service ruined everything." Is that positive or negative? A human would probably say "mixed." The model might say "positive" one time and "negative" the next, depending on which part it focused on. Same prompt, different answer. That is the risk with zero-shot.

Storyboard -- Zero-Shot · Works, but Edge Cases Wobble
PART C Few-Shot Prompting
Voiceover + Animation ~1.5 min

Strategy two: few-shot. You give the model a few examples before the actual task. "Here are three reviews with their classifications. Now classify this one." Suddenly the edge case gets consistent results. The model learns the pattern from your examples.

Storyboard -- Few-Shot · Show, Then Ask
PART D Chain-of-Thought -- The Bat and Ball
Voiceover + Animation ~2.5 min

Strategy three: chain-of-thought. You tell the model to think step by step. Here is the classic example. "A bat and a ball cost $1.10 together. The bat costs $1.00 more than the ball. How much does the ball cost?"

Without chain-of-thought, most models say 10 cents. That is wrong. With chain-of-thought -- "Think step by step" -- the model works through the algebra and gets 5 cents. Just four words can dramatically improve accuracy on reasoning tasks.

When do you use which? Zero-shot for simple tasks. Few-shot when you need consistent formatting. Chain-of-thought when the task requires reasoning. You will practice all three in Lab 1.

Video 4: "From Text to Structure"

Voiceover + animation + code -- 7-9 minutes -- 4 segments (A intro · B Problem with Text · C JSON Mode · D Pydantic)
KEY MESSAGE JSON mode guarantees parseable output. Pydantic guarantees correct structure.
PART A Intro — The Second Level of Control
Voiceover + Animation ~45 sec

So far in this module, you have learned how to talk to the model. You know how to set the role, the task, the context, the format. You know when to use examples and when to ask for step-by-step reasoning. That is the first level of control -- input control.

But here is the thing. Even when you ask the model beautifully, you still get back text. Free-form paragraphs. A block of writing. And that is fine when the answer is for a human to read. But the moment you want to do something with the output -- store it, filter it, feed it into another system -- text becomes a problem.

This video is about the second level of control. Not just what the model says, but the exact shape of what it hands you back. We will cover two tools: JSON mode, which forces the model to return valid JSON, and Pydantic, which makes sure that JSON has exactly the fields and types your code needs.

MEDIA TEAM: All three cards visible but Level 2 highlighted (amber). Level 1 shows a green ✓ check. Level 3 is faded out with a ⏭ "next video" marker. This card is a progress bar the student can see themselves moving through.
PART B The Problem with Text
Voiceover + Animation ~90 sec

Let me make the problem concrete. The output of a model, by default, is text. Free-form, unstructured text. And text is great for humans -- but terrible for code.

You cannot put a paragraph into a database column. You cannot feed prose into an analytics pipeline. You cannot reliably pull three numbers out of a sentence using string matching without it breaking the moment the wording changes. If your goal is to use the model's output in code, text is the enemy.

What you need instead is structure. You need to know, in advance, that the field called "sentiment" will always be there, and it will always be one of three values, and "confidence" will always be a number between zero and one. That is what JSON gives you -- a contract.

Storyboard -- Text Is Great for Humans, Bad for Code
PART C JSON Mode
Voiceover + Code Display ~2 min

JSON mode forces the model to return valid JSON -- guaranteed parseable, guaranteed structured.

Watch the difference. Same question, two modes.

Text mode: "The sentiment of the review is positive. The customer expressed satisfaction with the product quality and delivery speed."

JSON mode: {"sentiment": "positive", "confidence": 0.92, "keywords": ["quality", "delivery speed"]}

Same information. But the JSON version can be parsed by code in one line. No regex. No string manipulation. Just json.loads() and you have a Python dictionary.

Storyboard -- 12 Lines vs 1 Line
PART D Pydantic Validation
Voiceover + Animation ~2 min

But JSON mode has a gap. It guarantees valid JSON, but it does not guarantee the right fields or the right types. That is where Pydantic comes in.

Pydantic lets you define a schema -- a class that says "I expect a field called sentiment of type string, a field called confidence of type float between 0 and 1." If the model returns something that does not match, Pydantic catches it immediately. No silent bugs. No bad data reaching your database.

Video 6: "Function Calling -- From Text to Action"

Voiceover + animation + code -- 6-8 minutes -- 2 segments
KEY MESSAGE The model decides which function to call. Your code does the work. That is the bridge to agents.
PART A What Is Function Calling
Voiceover + Animation ~2.5 min

Until now, the model only produces text -- or structured text. Function calling lets it produce actions.

You define tools -- functions with names, descriptions, and parameter schemas. The model reads the user request and decides: which tool should I use? What arguments should I pass?

The model does not execute the function. It generates a structured call -- a JSON object saying "call this function with these arguments." Your code handles execution.

This is the bridge between a chatbot and an agent. The model reasons about which tool to use. Your code does the work.

Storyboard -- From Chatbot to Agent
PART B The Five-Step Weather Example
Voiceover + Step Animation ~3 min

Here is a complete example.

Step 1: You define a function called get_weather with parameters city and units.

Step 2: User says "What is the weather in Boston?"

Step 3: The model does not answer directly. Instead it returns: {"name": "get_weather", "arguments": {"city": "Boston", "units": "fahrenheit"}}

Step 4: Your code calls the real weather API with those arguments.

Step 5: You send the result back to the model. Now it generates a human-readable response: "It is currently 72 degrees in Boston with partly cloudy skies."

Five steps. The model decided which function to call. Your code did the work. That is function calling.
